## 一、核心业务问题定义

### 1.1 问题陈述

当前企业法务/合规团队在处理合同及法务文档时，面临**低效、易错、不可追溯**三重困境：

| 问题维度 | 具体表现 | 来源依据 |
|---------|---------|---------|
| **效率问题** | 人工逐字审查耗时极长，无法应对高频合同场景 | business-summary § 一 |
| **准确性问题** | 人工审查对隐性条款（不对称责任、稀释条款、跨条款冲突）容易遗漏 | business-summary § 一 |
| **可追溯性问题** | 现有工具缺乏完整审计日志，无法复现"谁在何时基于何依据做出判断" | analysis-summary § 六 |
| **决策透明度问题** | 审核结论缺乏来源、置信度、推理说明，用户无法判断是否应信任AI输出 | analysis-summary § 四 |

### 1.2 问题边界（硬约束）

> 以下约束直接来自业务调研的 Lead 仲裁结论，**不可在产品设计中绕开**：

1. **AI 定位**：辅助合规初审，不替代法律判断。系统输出的是"辅助识别结论"，最终决策权归属人工审核者。
2. **禁止 AI 自主判断的文档类型**：融资股权文件、跨司法管辖区竞业禁止条款有效性、合作框架与下游合同的冲突判断。
3. **置信度强制分级**：所有输出必须区分"事实提取（高置信）"和"法律解释（低置信）"，禁止混同展示。
4. **高风险强制 HITL**：风险等级 Critical/High、置信度 < 50%、未知文档类型，必须触发人工审核节点。
5. **Prompt Injection 防护**：原始文档内容不直接嵌入 System Prompt。

## 二、目标用户

### 2.1 主要用户角色

| 用户角色 | 核心诉求 | 使用场景 |
|---------|---------|---------|
| **企业法务 / 合规专员** | 高效完成合同初审，识别显性风险条款，减少重复性检索工作 | 日常合同流转审核 |
| **律师（法律顾问）** | 在 AI 标记的重点区域进行深度判断，审核 AI 结论的可靠性 | 高风险条款人工复核（HITL） |
| **业务负责人（非法律背景）** | 快速获得整体风险评分，判断是否需要升级法律资源介入 | 合同签约前快速决策 |

### 2.2 用户优先级

- **P0 核心用户**：企业法务 / 合规专员（系统的日常主要操作者）
- **P1 关键用户**：律师 / 法律顾问（HITL 审核节点的执行者，质量的最终保障）
- **P2 次要用户**：业务负责人（消费审核结论，不直接操作系统）

### 2.3 用户使用前提

- 用户具备基本的合同文档处理经验
- 法律专业用户（律师）能够理解置信度分级含义，不将 AI 输出等同于法律结论
- 企业具备法务审核的内部流程，本系统嵌入已有流程而非重建流程

## 三、MVP 业务场景总结

### 3.1 场景优先级原则

依据竞品分析的分阶段产品路线（analysis-summary § 六），MVP 聚焦可靠性与核心闭环，不追求功能完整性。

### 3.2 MVP 核心业务场景

---

#### 场景 A：标准化合同文档上传与解析

**触发条件**：用户上传合同文档（PDF / Word 格式）  
**核心流程**：
1. 用户上传文档（支持拖拽 / 本地选择）
2. 系统自动识别文档类型（采购合同 / NDA / 劳动合同 / SaaS 协议）
3. 文档解析：优先直接文本提取，扫描件升级 OCR（PaddleOCR），失败则降级人工
4. OCR 质量门控：扫描版文档须通过质量检测后才进入审核流程
5. 系统通过 WebSocket 推送实时进度（上传 → 解析 → 审核）

**成功标准**：
- 解析成功率 ≥ 90%（标准 PDF）
- 用户能实时看到处理状态，不需要刷新页面

**文档类型范围**（MVP 阶段）：
- ✅ 标准化采购合同
- ✅ NDA / 劳动合同（条款存在性检查，不判断有效性）
- ✅ SaaS 协议（全文送 LLM，不分块截断）
- ❌ 融资股权文件（不纳入 MVP）
- ❌ 合作框架协议（上下位条款冲突无法可靠处理，不纳入 MVP）

---

#### 场景 B：三层自动审核 + 风险识别

**触发条件**：文档解析成功后自动触发  
**核心流程**：

```
Layer 1（< 1s）：格式校验、元数据提取、文档分类
         ↓
Layer 2（3-10s）：条款识别、规则引擎匹配、风险初筛（置信度 ≥ 70% 输出结果）
         ↓
Layer 3（10-30s）：LLM 深度分析，复杂条款理解、跨条款关联
         ↓
    置信度 < 50% 或风险等级 High/Critical → 触发 HITL
```

**审核输出内容**（MVP 必须包含）：
- 整体风险评分（0-100，5 色阶）
- 分类风险列表（按条款类型 + 风险等级排序）
- 每条风险项：风险描述 + 原文定位 + 法规来源引用 + 置信度标注
- 事实提取字段（合同主体、金额、期限、关键日期）

**置信度分级规范**：
- 事实提取：高置信（≥ 90%），绿色标注
- 条款存在性检查：中置信（70-89%），黄色标注
- 风险评估 / 法律解释：低置信（< 70%），橙色标注 + 必须说明推理依据

---

#### 场景 C：HITL 人工审核节点

**触发条件**（任意一条满足）：
- 风险等级 = Critical 或 High
- AI 置信度 < 50%
- 文档类型未能识别
- 用户手动标记"质疑此结论"

**核心流程**：
1. 系统将任务状态流转至 `human_reviewing`
2. 分配给对应审核人（律师 / 法务专员）
3. 审核人在双视图界面查看：左侧原文 PDF（带高亮标注）+ 右侧 AI 审核结论
4. 点击风险条目 → 原文自动滚动并高亮对应位置（段落级精度，MVP）
5. 审核人可：确认 / 驳回 AI 结论、添加批注
6. 完成后状态流转至 `completed`

**超时管理**：
- 审核任务 > 30 分钟未处理 → 触发催办通知
- SLA 监控每 5 分钟检查一次超期任务

---

#### 场景 D：审核结果查阅与来源追溯

**触发条件**：审核完成后，任意用户查看报告  
**核心展示内容**：

```
Level 1 → 整体风险得分（色阶仪表盘）
Level 2 → 分类风险看板（条款类型 × 风险权重）
Level 3 → 风险列表（可按风险等级 / 条款类型过滤排序）
Level 4 → 单条款详情（置信度 + 原文高亮 + 推理说明 + 法规来源）
```

**来源引用规范**（MVP 必须实现）：
- 法规完整引用路径（如：《合同法》第 52 条第 3 款）
- 行内引用标注（上标形式）
- 来源折叠展开面板

**差异化功能（MVP 不实现）**：
- 风险热力图
- 决策树可视化
- 团队多角色协作批注

---

### 3.3 审核状态机（完整定义）

```
uploaded → parsing → parse_failed（终态）
         ↓
      parsed → auto_reviewing → auto_review_failed
                              ↓
                         auto_reviewed → [置信度高/低风险 → completed]
                                       ↓ [触发HITL条件]
                              human_reviewing → human_review_failed
                                             ↓
                                          completed（终态）

任意状态 → rejected（终态，含审核回退原因）
```

状态机是后续**数据模型设计的核心实体**，文档任务（ReviewTask）的 `status` 字段必须覆盖上述 10 个状态。

## 四、功能清单 → 业务建模映射

> 本节将功能清单升维为业务实体与核心关系，为系统架构和数据模型设计提供直接输入。

### 4.1 核心业务实体

| 实体 | 说明 | 关键属性 |
|------|------|---------|
| `Document` | 用户上传的原始文档 | 文件名、格式、大小、上传时间、解析状态、OCR质量分 |
| `ReviewTask` | 一次文档审核任务 | 文档ID、状态（10态）、审核类型、分配用户、创建/完成时间、SLA截止时间 |
| `RiskItem` | 单条风险识别结果 | 所属任务ID、风险类型、风险等级（5级）、置信度、置信度类别（事实/条款/法律）、原文定位（页码+段落）、推理说明 |
| `ClauseExtraction` | 结构化条款提取结果 | 所属任务ID、条款类型、原文内容、提取字段值（金额/日期/主体等）、置信度 |
| `SourceReference` | 来源引用记录 | 所属风险项ID、引用类型（法规/标准/案例）、引用路径、引用文本 |
| `HumanReview` | 人工审核操作记录 | 所属任务ID、审核人ID、操作类型（确认/驳回/批注）、操作时间、批注内容 |
| `User` | 系统用户 | 用户ID、角色（法务/律师/业务方）、所属组织 |

### 4.2 核心业务关系

```
User (1) ──────── (N) ReviewTask          # 用户发起审核任务
Document (1) ───── (1) ReviewTask         # 一份文档对应一个主审核任务
ReviewTask (1) ─── (N) RiskItem           # 一个任务产生多条风险项
ReviewTask (1) ─── (N) ClauseExtraction   # 一个任务产生多条条款提取
RiskItem (1) ────── (N) SourceReference   # 每条风险项可有多个来源引用
ReviewTask (1) ─── (N) HumanReview        # 人工审核任务可有多条操作记录
User (1) ──────── (N) HumanReview         # 审核人操作记录
```

### 4.3 关键业务规则（架构设计必须遵守）

1. **置信度类别与展示强绑定**：`RiskItem.confidence_category` 决定 UI 展示色彩和警示等级，不可由前端自行计算。
2. **状态流转单向性**：审核状态只能按状态机定义的路径流转，禁止任意跳转（`completed` 无法回退到 `auto_reviewing`）。
3. **HITL 触发是系统行为，非用户行为**：HITL 触发条件在后端判断，前端无权绕过。
4. **审计日志全量记录**：所有状态变更、人工操作必须写入不可变日志，支持合规审计追溯。
5. **向量库版本绑定**：每份 `ReviewTask` 必须记录审核时所用向量库版本，支持事后重现和审计。

## 五、MVP 范围边界总结

### 纳入 MVP

| 能力 | 优先级 |
|------|--------|
| 多格式文档上传（PDF/Word）+ WebSocket 实时进度 | P0 |
| 多引擎文档解析（直取→OCR→降级人工）+ OCR 质量门控 | P0 |
| 三层自动审核 + 置信度分级输出 | P0 |
| HITL 人工审核节点（双视图 + 段落级原文定位） | P0 |
| 10 状态审核状态机 + 超时催办 | P0 |
| 4 层递进风险信息架构展示 | P0 |
| 法规来源引用标注 | P0 |
| 审计日志（状态变更 + 人工操作） | P0 |
| 事实字段结构化提取（金额/日期/主体） | P1 |
| AI 推理自然语言说明 | P1 |

### 排除 MVP

| 能力 | 排除原因 |
|------|---------|
| 融资股权文件审核 | 错误代价极高，AI 可行性极低 |
| 合作框架合同联动审查 | RAG 无法可靠处理上下位条款冲突 |
| 竞业禁止有效性判断 | 司法管辖区差异导致不可靠 |
| 风险热力图 | 竞品分析定位为 V2 功能 |
| 团队多角色协作批注 | 竞品分析定位为 V2 功能 |
| 决策树可视化 | 竞品分析定位为 V2 功能 |
| 多角色权限体系 | 竞品分析定位为 V2 功能 |
