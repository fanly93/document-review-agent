## 一、调研背景与问题定义

### 1.1 问题描述

基于业务调研汇总的结论，合同/法务文档 AI 审核系统需要建设一套**完整的流程架构**，涵盖从用户上传文档开始，到自动审核、人工审核、最终输出的全生命周期。该流程需要满足：

- **高可靠性**：处理法律级别的审核意见，错误不可承受
- **全可追踪性**：每个审核决策都需要审计日志和决策链路
- **人机协作**：AI 自动审核与人工审核的无缝衔接
- **高效率**：降低律师的重复性工作，加快初审效率
- **错误恢复**：支持审核回退、修改、重新处理

### 1.2 调研范围

本调研聚焦于以下五大核心流程环节：

1. **文件上传流程**：格式支持、大小限制、批量处理、进度反馈、错误处理
2. **文档解析处理**：OCR识别、版面分析、表格解析、多格式支持
3. **Agent自动审核**：AI模型调用链、规则引擎、流程设计、置信度机制
4. **人工审核流程**：任务分配、队列管理、人机协作、意见录入
5. **审核状态管理**：状态机设计、流转规则、超时处理、回退机制

## 三、核心流程分析

### 3.1 文件上传流程对比

#### 3.1.1 ABBYY FineReader 方案

**上传策略**：
- 支持格式：PDF、Word、Excel、图片（JPG、PNG、TIFF）、扫描版文档
- 单文件限制：最大 500MB（网页版受限，桌面版可更大）
- 批量上传：支持，但采用队列制，需等待前序处理完成
- 上传进度：基础百分比条，无细粒度分段反馈

**错误处理**：
- 格式校验：上传前端验证 + 服务端二次校验
- 文件损坏检测：自动检测损坏PDF（无法修复则拒绝）
- 重试机制：服务端自动重试 3 次，用户可手动重新上传
- 超时处理：30 分钟内未完成则超时失败，不支持断点续传

**典型交互**：
```
用户选择文件 → 前端校验格式/大小 → 上传 → 扫描病毒 → 入库 
→ 触发 OCR 队列 → 用户轮询状态
```

#### 3.1.2 AWS Textract 方案

**上传策略**：
- 支持格式：PDF（单页/多页）、PNG、JPEG、GIF、WEBP
- 单文件限制：PDF 最大 500 页或 100MB，图片最大 10MB
- 批量上传：支持 Batch 操作，单批最多 100 个文档
- 上传进度：S3 预签名 URL 支持断点上传，支持多线程并行

**错误处理**：
- 格式校验：严格检查，不支持的格式直接拒绝
- 页面级别处理：单页失败不影响其他页
- 重试策略：自动重试失败的页面，可配置重试次数
- 超时处理：异步处理，支持长期 Job 轮询

**典型交互**：
```
用户点击上传 → 获取 S3 预签名 URL → 浏览器直传 S3 
→ 触发 Textract Job → 用户通过 Job ID 轮询结果
```

**优势**：断点续传、多线程上传、页面级粒度错误处理

#### 3.1.3 合合信息 Datawhale 方案

**上传策略**：
- 支持格式：PDF、Word、Excel、JPG/PNG、扫描件
- 单文件限制：PDF 最大 200MB，智能分批处理超大文件
- 批量上传：支持，采用后端队列自动编排，用户可上传后即可离开
- 上传进度：WebSocket 实时推送处理进度（分阶段：上传→OCR→提取）

**错误处理**：
- 格式校验：自动识别文件类型，纠正扩展名错误
- OCR 失败恢复：自动尝试多个 OCR 引擎，若全部失败则降级为手动认领
- 重试机制：后端自动重试，用户无需干预
- 超时处理：支持离线处理，可在后台异步运行数小时

**典型交互**：
```
用户拖拽上传 → 后端自动识别格式 → 智能分批 
→ WebSocket 推送：上传进度 → OCR 进度 → 提取进度
→ 全部完成后用户进入审核界面
```

**优势**：全自动化、实时反馈、无需主动轮询

#### 3.1.4 Azure Form Recognizer 方案

**上传策略**：
- 支持格式：PDF、PNG、JPEG、BMP、TIFF、GIF
- 单文件限制：单页最大 4MB，PDF 最多 2000 页
- 批量上传：通过 Batch API，支持异步处理多个文件
- 上传进度：无原生进度反馈，需要应用层实现

**错误处理**：
- 格式校验：严格，预训练模型对格式敏感
- 页面级处理：支持指定处理页面范围（from/to）
- 重试策略：应用层可实现，SDK 无内置重试
- 超时处理：单次请求 10 分钟超时

**限制**：API 层面缺乏进度反馈和细粒度重试机制

#### 3.1.5 Google Document AI 方案

**上传策略**：
- 支持格式：PDF、GIF、PNG、JPEG、WEBP、TIFF、BMP
- 单文件限制：单次请求 20MB，但支持 Batch 处理任意大小
- 批量上传：Batch Process API，单批最多 1000 个文件
- 上传进度：Batch 操作支持进度查询

**错误处理**：
- 格式校验：自动格式检测，兼容性强
- 页面级处理：每页独立处理，失败隔离
- 重试策略：内置自动重试，可配置重试策略
- 超时处理：支持长期 Batch Job，最长 7 天

**特性**：支持 GCS 直连，与 Cloud Storage 整合紧密

#### 3.1.6 DocuWare 企业工作流方案

**上传策略**：
- 支持格式：PDF、Word、Excel、原生格式（自动转 PDF）
- 单文件限制：可配置，通常 1GB+
- 批量上传：支持文件夹批量导入，自动拆分和编号
- 上传进度：企业级监控面板，提供详细进度统计

**错误处理**：
- 格式校验：宽松，自动转换不支持格式
- 失败处理：失败文件自动归档，管理员可干预
- 重试机制：企业级策略配置，可设置无限重试
- 超时处理：后台队列处理，支持跨天续传

**优势**：企业级可靠性、完整的监控和管理面板

### 3.2 文档解析处理对比

#### 3.2.1 ABBYY FineReader 文档解析

**OCR 能力**：
- 精度：98%+ 黑白文本，94-96% 彩色/混合版

- 特殊处理：手写识别、数学公式识别、条形码识别
- 速度：典型 A4 页 1-3 秒（CPU 模式）

**版面分析**：
- 页面布局检测：自动识别文本区域、表格、图片、页脚
- 逻辑阅读顺序：恢复自然阅读流
- 多栏检测：自动识别多栏布局，正确排序

**表格识别**：
- 结构识别：单表识别率 95%+，嵌套表格支持度 80-85%
- 内容提取：保留单元格格式、合并单元格
- 输出格式：CSV、Excel、XML、HTML

**图片处理**：
- 预处理：自动旋转、倾斜校正、去噪、灰度优化

- 图表识别：基础支持，不提取结构化数据

**工作流**：
```
原始文件 → 预处理（旋转/倾斜/去噪）
→ 版面分析（区域检测）
→ 分区块处理（文本/表格/图片）
→ 分块 OCR → 后处理（结构恢复）
→ 输出（PDF/Excel/CSV）
```

### 3.3 Agent 自动审核流程对比

#### 3.3.1 合合信息 Datawhale AI 审核流程

**架构设计**：
```
上传的合同
  ↓
文档解析（OCR + 版面分析）
  ↓
关键字段自动提取（合同方、金额、期限等）
  ↓
并行审核：
  ├─ 条款完整性检查（对标模板库）
  ├─ 风险条款识别（基于规则 + NLP）
  ├─ 数字字段校验（金额、日期等）
  └─ 法规合规性检查（地域、法律主体）
  ↓
置信度评分与聚合
  ↓
人工审核队列路由
```

**关键特性**：

| 特性 | 实现方式 |
|-----|--------|
| **知识库管理** | 维护行业标准条款库（采购/劳动/SaaS）、法规库、风险规则库 |
| **规则引擎** | 基于正则 + 关键词匹配，可配置的风险规则 |
| **NLP 能力** | 法律领域 BERT 模型，用于条款理解和相似度匹配 |
| **置信度** | 四级：高（95%+）、中（70-95%）、低（50-70%）、待确认（<50%） |
| **AI 建议** | 风险等级 + 修改建议（模板驱动） |
| **审核回溯** | 每个判断都可溯源到具体条款和规则 |

#### 3.5.2 Google Document AI + Firestore 状态模型

  ├─ PARSED: 解析完成
  ├─ AI_REVIEWING: AI 审核中
  ├─ AI_REVIEW_COMPLETE: AI 审核完成
  ├─ HUMAN_REVIEW_PENDING: 等待人工审核
  ├─ HUMAN_REVIEWING: 人工审核中
  ├─ HUMAN_REVIEW_COMPLETE: 人工审核完成

  UPLOADED → PARSING

  PARSED → AI_REVIEWING
  AI_REVIEWING → AI_REVIEW_COMPLETE

#### 3.5.4 LangChain 状态机设计（参考）

**使用 LangGraph 定义状态**：
```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph

class DocumentState(TypedDict):
    doc_id: str
    raw_text: str
    status: str  # uploaded, parsing, parsed, ai_reviewing, human_reviewing, completed
    parsed_content: dict
    ai_findings: list
    human_review: dict
    current_step: int
    error_message: str | None
    metadata: dict

# 定义节点函数
def parse_document(state: DocumentState):
    # 调用文档解析服务
    return {
        "parsed_content": ...,
        "status": "parsed"
    }

def ai_review(state: DocumentState):
    # 调用 Agent 进行 AI 审核
    return {
        "ai_findings": ...,
        "status": "ai_review_complete"
    }

def human_review_decision(state: DocumentState):
    # 判断是否需要人工审核
    risk_level = state["ai_findings"]["risk_level"]
    if risk_level == "high":
        return "human_review"
    else:
        return "approve"

# 构建状态图
graph = StateGraph(DocumentState)
graph.add_node("parse", parse_document)
graph.add_node("ai_review", ai_review)
graph.add_node("human_review", human_review)
graph.add_node("approve", approve_document)

graph.add_edge("parse", "ai_review")
graph.add_conditional_edges(
    "ai_review",
    human_review_decision,
    {"human_review": "human_review", "approve": "approve"}
)
```

**优势**：
- 状态显式定义，易于追踪
- 支持复杂的条件路由和并行处理
- 与 LangSmith 集成，全链路可观测

#### 3.5.5 超时和回退处理

**超时处理**：
```
定时任务（每 5 分钟执行）：
  ├─ 查询所有 status = "HUMAN_REVIEWING" 且超过 SLA 的文件
  ├─ 为过期文件更新 is_overdue = true
  ├─ 发送通知给审核人
  ├─ 发送催办给主管
  └─ 若连续 2 小时未更新，自动将文件返回队列供其他人审核
```

**回退机制**：
```

  ├─ 状态: HUMAN_REVIEWING → MODIFICATION_REQUIRED
  ├─ 创建回退记录（谁、什么时间、原因）
  ├─ 发送通知给原提交者
  ├─ 文件从审核队列移出
  ├─ 原提交者修改后重新提交
  └─ 重新进入审核队列（优先级不变）

回退链路可追踪：
  /documents/{doc_id}/review_history/
    ├─ round_1: { reviewer, status, timestamp, notes }
    ├─ round_2: { ... }
    └─ ...
```

### 4.3 不同架构的对比

#### 传统工作流方案（如 DocuWare）
**优点**：
- 流程管理完整，人机协作界面成熟
- 合规性强，审计日志完善
- 部署和维护相对简单

**缺点**：
- AI 能力弱，难以深度融合
- 定制化能力受限（低代码）
- 无法处理非结构化文本的智能分析

#### 云函数 + 规则引擎方案（如 AWS Lambda）
**优点**：
- 成本低，自动扩展
- 无服务管理，运维简单
- 集成 AWS 其他服务方便

**缺点**：
- 规则引擎表达能力有限，难以处理复杂推理
- 缺乏灵活的人机协作界面
- 可观测性弱，调试困难

#### LLM Agent + 工作流方案（LangChain）
**优点**：
- 灵活性最高，可处理复杂推理和非结构化理解
- 可观测性强，完整链路追踪
- 支持 HITL，人机协作自然

**缺点**：
- 开发和运维成本高
- Token 消耗多（成本昂贵）
- LLM 幻觉问题，需要严格的约束机制
- 法律应用中，错误代价高

#### 混合方案（推荐）
```
高价值区 → LLM Agent (灵活推理)
低价值区 → 规则引擎 (快速、确定)
工作流管理 → 自定义工作流引擎（不依赖第三方低代码平台）
人机协作 → 轻量级 UI 框架（React/Vue）
```

3. **批量上传编排**
   - 支持文件夹上传，自动扫描子目录
   - 后端自动分批，避免单批过大
   - 为每个文件生成唯一 ID，支持后续追踪

4. **错误处理与恢复**
   - 上传前端检验 + 后端二次检验
   - 自动检测文件损坏、加密、损坏 PDF
   - 失败文件保存状态，支持重新上传而无需重新选择
   - 病毒扫描可选（生产环境建议启用）

### 5.2 文档解析建议

**推荐架构**：
```
多引擎分层：
  ├─ 第一层（中文优先）：PaddleOCR 或合合信息 API
  ├─ 第二层（兜底）：ABBYY 或 Textract
  ├─ 版面分析：Google Document AI Layout Processor
  └─ 表格识别：Textract Table Extraction（精度最高）
```

**关键指标**：
- 中文 OCR 精度目标：99%+
- 英文混合精度目标：98%+
- 表格识别准确率：96%+
- 单页平均处理时间：<2 秒

**分块策略**：
- 按条款边界分块（针对合同）
- 保留跨块引用元数据
- 为每个分块保留原始位置坐标和页码
- 支持版面结构信息（标题、段落、列表、表格层级）

### 5.3 Agent 自动审核流程建议

**分层审核策略**：

```
Layer 1: 快速检查（规则引擎，<1 秒）
  ├─ 文件格式完整性：检测必需字段是否识别成功
  ├─ OCR 质量评估：置信度均值 <60% 则标记为低质
  └─ 基础字段提取：合同方、金额、期限

Layer 2: 结构化检查（LLM Agent，3-10 秒）
  ├─ 字段值合理性：日期是否合理、金额是否在范围内
  ├─ 条款完整性：对照标准条款库检查缺失的关键条款
  ├─ 条款冲突检测：跨条款矛盾识别
  └─ 初步风险评分：低/中/高/非常高

Layer 3: 深度分析（可选，仅高风险，10-30 秒）
  ├─ 免责条款合法性：基于法律知识库评估
  ├─ 潜在歧义识别：语义不清的表述
  ├─ 竞业禁止有效性：按地域法律评估
  └─ 细粒度风险分类
```

**置信度管理**：
```
confidence_score = {
  "fact_extraction": 0.95,  // 字段提取：高置信
  "clause_completeness": 0.70,  // 条款完整性：中置信
  "risk_assessment": 0.50   // 风险评估：低置信（不应输出确定结论）
}

输出规则：
  - fact_extraction > 0.90：直接输出
  - 0.70 < clause_completeness < 0.90：输出但标记为"需人工确认"
  - risk_assessment < 0.70：仅输出为"参考"，加注"由律师最终判断"
```

**HITL 触发阈值**：
```
if overall_risk_level == "高" or "非常高":
  → 自动路由人工审核

elif overall_risk_level == "中" and low_confidence_findings > 3:
  → 路由人工审核

elif document_type not in ["采购合同", "SaaS", "劳动合同"]:
  → 路由人工审核（未经培训的文档类型）

else:
  → 自动通过（提供 AI 建议供审核人参考）
```

### 5.4 人工审核流程建议

**任务分配规则**：
```
优先级计算：
  priority = risk_level * 10 + aging_days * 2 + (is_overdue ? 100 : 0)

分配算法：
  1. 排序：按 priority 倒序
  2. 筛选：选择空闲且有审核权限的人
  3. 均衡：选择当前任务数最少的人
  4. 专长匹配：优先分配给审核过同类文档的人

特殊规则：
  - 高风险（非常高）：仅分配给资深律师
  - VIP 客户：优先级加倍
  - 跨部门审核：需要审核后转派确认
```

**人机协作界面建议**：
```
左侧（60%）：
  ├─ 原始文档查看器（PDF 渲染）
  ├─ 支持页面导航、缩放、搜索
  ├─ 显示页码和 AI 标记位置

右侧（40%）：
  ├─ AI 建议摘要
  │  ├─ 高置信发现（字段提取）
  │  ├─ 中置信发现（条款完整性）
  │  └─ 低置信发现（风险评估）参考
  ├─ 审核意见面板
  │  ├─ 风险等级选择（低/中/高/非常高）
  │  ├─ 预定义意见列表（可搜索）
  │  ├─ 自由文本输入
  │  └─ 关联到具体页码/条款
  └─ 操作按钮
     ├─ 保存草稿
     ├─ 提交完成
     ├─ 转派
     ├─ 返回修改（含反馈给修改方）
     └─ 询问 AI（实时追问）
```

**审核意见结构**：
```json
{
  "finding_id": "FIND_2026_001_001",
  "type": "clause_issue" | "missing_clause" | "risk_clause" | "data_error",
  "severity": "critical" | "high" | "medium" | "low",
  "related_clause": "条款 5.1",
  "location": { "page": 1, "start_char": 100, "end_char": 200 },
  "description": "...",
  "suggestion": "...",
  "requires_modification": true,
  "notes": "律师的补充说明"
}
```

### 5.5 审核状态管理建议

**推荐状态机**（10 个状态）：
```
UPLOADED 
  → PARSING 
    → PARSE_FAILED | PARSED
  
PARSED 
  → AI_REVIEWING 
    → AI_REVIEW_COMPLETE

AI_REVIEW_COMPLETE 
  → HUMAN_REVIEW_PENDING (自动路由或用户指定)
    → HUMAN_REVIEWING
      → HUMAN_REVIEW_COMPLETE | MODIFICATION_REQUIRED

MODIFICATION_REQUIRED 
  → RESUBMITTED 
    → HUMAN_REVIEW_PENDING (循环)

HUMAN_REVIEW_COMPLETE 
  → APPROVED | REJECTED

APPROVED | REJECTED 
  → ARCHIVED
```

**关键字段**：
```python
class DocumentStatus(TypedDict):
    status: str  # 上述 10 个状态之一
    current_step: str  # "parsing" | "ai_review" | "human_review"
    priority: str  # "low" | "normal" | "high" | "urgent"
    
    # 时间追踪
    created_at: datetime
    parsing_start_time: datetime | None
    parsing_end_time: datetime | None
    ai_review_start_time: datetime | None
    ai_review_end_time: datetime | None
    human_review_assigned_at: datetime | None
    human_review_start_time: datetime | None
    human_review_end_time: datetime | None
    
    # SLA
    sla_deadline: datetime
    is_overdue: bool
    
    # 分配与链路
    assigned_reviewer: str | None
    review_history: List[ReviewRecord]  # 每轮审核的记录
    modification_count: int
    
    # 审核结果
    ai_findings: List[Finding]
    human_findings: List[Finding]
    final_decision: "approved" | "rejected" | None
```

**状态转移事件**（可观测性）：
```
每次状态变化都记录：
  {
    "timestamp": "2026-04-14T14:30:00Z",
    "from_status": "PARSED",
    "to_status": "AI_REVIEWING",
    "triggered_by": "system" | "user_id",
    "reason": "automatic_routing" | "user_triggered",
    "metadata": { ... }
  }
```

**超时与催办**：
```
定时任务（每 5 分钟）：
  1. 找出所有 status = HUMAN_REVIEWING 且距 sla_deadline < 30 分钟的
     → 发送通知给审核人："XX 合同即将超期"
  
  2. 找出所有距 sla_deadline < 0 的
     → is_overdue = true
     → 发送告急通知给主管
     → 自动重新分配给其他人（可选）

回退处理：
  modification_count > 3 时：
    → 发送告警给审核主管
    → 分析修改原因，可能需要改进审核规则
```

## 六、总结与后续建议

### 6.1 关键设计原则

1. **分层处理**：快速路径（规则） + 标准路径（LLM） + 深度分析（可选）
2. **置信度管理**：明确区分"事实提取"（高置信）与"法律解释"（低置信）
3. **人机协作**：AI 辅助，人工最终决策，支持双向反馈
4. **完整可追踪**：所有审核决策都有审计日志，支持历史回溯
5. **容错恢复**：支持修改、回退、重新处理，无死路

### 6.2 优先级建议

| 优先级 | 功能 | 理由 |
|-------|------|------|
| P0 | 文件上传 + 文档解析 | 基础能力，直接影响后续所有步骤 |
| P0 | 状态管理 + 队列 | 工作流基座，支持人机协作 |
| P1 | AI 自动审核（Layer 1-2） | 核心价值，提高效率 |
| P1 | 人工审核界面 + 意见管理 | 人机协作关键 |
| P2 | AI 深度分析（Layer 3） | 可选，先验证 Layer 1-2 效果 |
| P2 | SLA 监控 + 催办 | 运营效率 |
| P3 | 报表与分析 | 数据驱动改进 |

### 6.3 技术栈建议

| 组件 | 推荐选择 | 理由 |
|------|--------|------|
| 文档解析 | PaddleOCR + Google Doc AI + Textract | 多引擎兜底，精度最优 |
| AI 审核 | LangChain + Claude/GPT-4 | 灵活性最高，推理能力强 |
| 工作流 | 自定义状态机（LangGraph） | 不依赖第三方低代码平台 |
| 队列管理 | Redis + Celery 或 Cloud Tasks | 可扩展，易于监控 |
| 数据存储 | PostgreSQL + Firestore | OLTP + 实时查询 |
| 前端 | React + Ant Design | 成熟、完整的企业级 UI |
| 可观测性 | LangSmith + 自定义日志 | 链路追踪 + 审计日志 |
