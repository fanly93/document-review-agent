## 一、文档目的与范围

### 1.1 目的

本文档聚焦于审核状态机中**六个中间过渡状态**的完整交互处理规范：

| 覆盖状态 | 状态语义 |
|---------|---------|
| `auto_reviewing` | AI 自动审核进行中 |
| `auto_review_failed` | AI 自动审核异常失败 |
| `parse_failed` | 文档解析失败（含 OCR 失败） |
| `human_reviewing` | 人工审核进行中（HITL 节点） |
| `human_review_failed` | 人工审核异常失败 |
| `rejected` | 审核任务被拒绝（任意状态可流转至此） |

本文档**不涉及**以下状态（由其他交互文档负责）：
- `uploaded`、`parsing`、`parsed`（上传与解析交互链路）
- `auto_reviewed`（HITL 触发判断逻辑）
- `completed`（审核结果展示）

### 1.2 设计原则

依据竞品分析关键洞察（analysis-summary § 六）：

1. **透明度优先**：每个中间状态必须向用户传达"当前发生了什么 / 为什么"，不可仅展示 loading。
2. **可操作性优先**：失败态必须提供明确的恢复路径，用户不应面对死路。
3. **不中断信任链**：状态流转信息、操作日志必须完整保留，支持事后审计（对应 problem_modeling § 四.4.3 关键业务规则）。


## 三、auto_reviewing — AI 自动审核进行中

### 3.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | `parsed`（文档解析成功） |
| **触发条件** | 文档解析通过 OCR 质量门控后，系统自动触发三层审核流程 |
| **持续时长** | 约 15–40 秒（Layer 1 < 1s + Layer 2 3-10s + Layer 3 10-30s） |
| **后续状态** | `auto_reviewed`（审核完成）/ `auto_review_failed`（异常失败） |

### 3.2 展示规范

#### 进度条与分阶段提示

由于自动审核分三层执行（problem_modeling § 三.2 场景 B），界面须将三层进度拆开展示，禁止使用单一不可区分的旋转 loading：

| 阶段 | 展示文案 | 预计时长 | 完成判断 |
|------|---------|---------|---------|
| Layer 1 | "格式校验 & 文档分类中…" | < 1s | 后端推送阶段完成事件 |
| Layer 2 | "条款识别 & 规则匹配中…" | 3–10s | 后端推送阶段完成事件 |
| Layer 3 | "LLM 深度分析中…" | 10–30s | 后端推送阶段完成事件 |

进度信息通过 **WebSocket** 实时推送（analysis-summary § 三.1），禁止前端轮询。

#### 视觉规范

- 整体进度：线性进度条（0–100%），根据各层预估时长分配权重（Layer 1: 5% / Layer 2: 35% / Layer 3: 60%）
- 当前层：旋转图标 + 蓝色高亮文案
- 已完成层：绿色勾选图标 + 灰色文案
- 预计剩余时间：在进度条下方以灰色小字展示（"预计还需约 XX 秒"）

#### 禁止行为

- ❌ 不得在 `auto_reviewing` 状态下展示任何部分审核结论（避免用户误读未完成结果）
- ❌ 不得允许用户在此状态下发起"重新上传"等覆盖操作

### 3.3 用户可执行操作

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 查看文档基本信息（文件名、大小、类型） | ✅ 允许 | 静态信息不依赖审核结果 |
| 取消本次审核任务 | ✅ 允许 | 流转至 `rejected`，需二次确认弹窗 |
| 查看其他已完成任务 | ✅ 允许 | 允许跳转，不影响当前任务 |
| 修改文档 / 重新上传 | ❌ 禁止 | 须等待当前任务完成或取消后操作 |
| 手动触发人工审核 | ❌ 禁止 | HITL 触发是系统行为（problem_modeling § 四.4.3） |

### 3.4 出口条件

| 出口 | 触发条件 | 跳转目标 |
|------|---------|---------|
| 正常完成 | 三层审核全部完成，后端返回 `auto_reviewed` 事件 | → `auto_reviewed` 状态页 |
| 系统异常 | 任一层发生不可恢复错误，后端返回 `auto_review_failed` 事件 | → `auto_review_failed` 状态页 |
| 用户主动取消 | 用户点击"取消审核"并确认 | → `rejected` 状态页 |

### 3.5 向下一状态的数据传递内容

流转至 `auto_reviewed` 时，需携带以下字段供后续处理使用：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 审核任务唯一标识 |
| `document_id` | UUID | 关联文档标识 |
| `layer1_result` | object | 格式校验结论、文档分类结果、元数据 |
| `layer2_result` | object | 条款识别列表、规则匹配结果、风险初筛列表 |
| `layer3_result` | object | LLM 深度分析结论、条款关联关系 |
| `overall_risk_score` | int (0-100) | 整体风险评分 |
| `risk_items` | array | 风险条目列表（含置信度、风险等级、原文定位） |
| `hitl_required` | boolean | 是否触发 HITL（由后端判断） |
| `hitl_trigger_reasons` | array | HITL 触发原因列表（如：风险等级Critical、置信度<50%） |
| `vector_store_version` | string | 本次审核使用的向量库版本（审计用途） |
| `completed_at` | timestamp | 自动审核完成时间 |

## 四、auto_review_failed — AI 自动审核异常失败

### 4.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | `auto_reviewing` |
| **触发条件** | 自动审核三层流程中任一层发生不可恢复的系统错误（网络超时、模型服务不可用、内部异常等） |
| **注意区分** | 此状态是**系统异常**，不同于"AI 审核出结论但置信度低"（后者正常流转至 `auto_reviewed`） |
| **后续状态** | `auto_reviewing`（重试成功）/ `human_reviewing`（降级人工）/ `rejected`（用户放弃） |

### 4.2 展示规范

#### 错误信息展示

- **标题**：使用明确的失败语义，如"AI 审核未能完成"，禁止使用模糊的"出错了"
- **说明文案**：简洁说明失败原因（按错误类型区分）：

| 错误类型 | 用户可见文案 | 是否展示技术细节 |
|---------|------------|--------------|
| 模型服务超时 | "AI 审核服务响应超时，请稍后重试" | 否（记录到审计日志） |
| 模型服务不可用 | "AI 审核服务暂时不可用，建议转为人工审核" | 否 |
| 文档内容异常（解析后发现无可审核内容） | "文档内容无法被有效提取，请检查文档完整性" | 否 |
| 未知内部错误 | "系统遇到未预期的问题，已记录，请选择重试或转人工" | 否 |

- **状态图标**：橙色警告图标（区别于 `rejected` 的红色 X 图标）
- **失败时间戳**：展示失败发生时间，支持用户联系客服时提供定位信息

#### 禁止行为

- ❌ 不得在界面展示原始错误堆栈或技术错误码（安全风险）
- ❌ 不得自动静默重试超过 1 次而不通知用户（避免用户感知不到系统状态）

### 4.3 用户可执行操作

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 重新发起 AI 审核 | ✅ 允许（主要操作） | 重试前显示"重试可能需要 30-60 秒"提示，流转回 `auto_reviewing` |
| 转为人工审核 | ✅ 允许（次要操作） | 跳过 AI 审核，直接进入 `human_reviewing`；须有权限才可操作 |
| 取消任务 | ✅ 允许 | 流转至 `rejected` |
| 下载原始文档 | ✅ 允许 | 用户可取回文档，不依赖审核结果 |
| 查看部分已完成层的结果 | ❌ 禁止 | 部分结果不可靠，展示会误导用户决策 |

### 4.4 出口条件

| 出口 | 触发条件 | 跳转目标 |
|------|---------|---------|
| 重试成功 | 用户点击重试，后端重新执行审核成功 | → `auto_reviewing` → `auto_reviewed` |
| 转人工 | 用户点击"转为人工审核" | → `human_reviewing` |
| 用户放弃 | 用户点击"取消任务" | → `rejected` |

### 4.5 向下一状态的数据传递内容

**流转至 `auto_reviewing`（重试）时**，需携带：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 保持同一任务标识，不创建新任务 |
| `retry_count` | int | 已重试次数（用于限制最大重试次数） |
| `failed_at_layer` | string | 失败发生的层级（layer1/layer2/layer3） |
| `failure_reason` | string | 内部错误原因（不展示给用户，记录日志） |
| `failure_timestamp` | timestamp | 失败时间（审计用途） |

**流转至 `human_reviewing`（转人工）时**，需携带：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 审核任务标识 |
| `document_id` | UUID | 关联文档标识 |
| `escalation_reason` | string | 升级原因，固定值："auto_review_failed" |
| `partial_layer_results` | object | 失败前已完成层的结果（若有），供人工审核参考 |
| `assigned_reviewer_id` | UUID | 分配的人工审核员 ID（由系统或管理员指定） |
| `sla_deadline` | timestamp | SLA 截止时间（从升级时刻起算） |

## 五、parse_failed — 文档解析失败

### 5.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | `parsing`（解析进行中） |
| **触发条件** | 文档经直接文本提取 → OCR → 降级人工三层降级策略后，仍无法获得可用文本；或 OCR 质量检测不达标 |
| **终态标志** | `parse_failed` 是**终态**（problem_modeling § 三.3 状态机定义），不自动重试 |
| **后续状态** | 用户重新上传（创建新任务）/ `rejected`（主动放弃） |

### 5.2 展示规范

#### 失败原因分类展示

| 失败原因 | 用户可见文案 | 建议操作引导 |
|---------|------------|------------|
| 文件损坏 / 无法读取 | "文档无法打开，可能文件已损坏" | 建议用户重新导出文档 |
| 扫描件 OCR 质量不达标 | "扫描件清晰度不足，无法识别文字内容" | 建议提供更高分辨率版本（≥ 300 DPI） |
| 不支持的格式 | "当前文件格式暂不支持（支持 PDF / Word）" | 展示支持格式列表 |
| 加密 / 受密码保护 | "文档受密码保护，请先移除密码后重新上传" | 提供移除密码的操作提示 |
| 纯图片 PDF 且 OCR 失败 | "文档为纯图片格式，自动识别失败" | 建议转人工录入或提供可编辑版本 |

#### 视觉规范

- **状态图标**：红色 X 图标（终态）
- **不展示进度条**：终态无需进度信息
- **文档缩略图**：展示上传文件的基本信息（文件名、大小、上传时间），让用户确认是否操作了正确的文件

### 5.3 用户可执行操作

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 重新上传文档（创建新任务） | ✅ 允许（主要操作） | 跳转至上传页面，本次任务保持 `parse_failed` 终态不变 |
| 下载失败前的原始文件 | ✅ 允许 | 让用户取回文件，确认是否上传了正确文件 |
| 查看失败原因详情 | ✅ 允许 | 展开详细说明面板（格式要求、清晰度要求等） |
| 重试解析（同一文件） | ❌ 禁止 | 终态不可重试（problem_modeling § 四.4.3 状态流转单向性） |
| 直接转人工审核 | ❌ 禁止 | 文档内容不可用，人工审核也无法操作 |

### 5.4 出口条件

| 出口 | 触发条件 | 跳转目标 |
|------|---------|---------|
| 重新上传 | 用户点击"重新上传"，选择新文件 | → 新任务 `uploaded` 状态 |
| 放弃 | 用户离开页面或点击"放弃" | 当前任务保持 `parse_failed` 终态 |

### 5.5 向下一状态的数据传递内容

由于 `parse_failed` 是终态，不向其他状态流转。但若用户选择"重新上传"创建新任务，需携带以下上下文信息（用于改善用户体验）：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `original_task_id` | UUID | 原失败任务的 ID（用于关联追溯） |
| `failure_reason_code` | string | 失败原因枚举（用于新任务页面的提示预填） |
| `document_type_hint` | string | 系统猜测的文档类型（若有，预填给用户确认） |

## 六、human_reviewing — 人工审核进行中（HITL 节点）

### 6.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | `auto_reviewed`（HITL 触发判断后）/ `auto_review_failed`（AI 失败降级） |
| **触发条件**（系统自动判断，非用户行为） | 风险等级 Critical/High / AI 置信度 < 50% / 文档类型未识别 / 用户手动标记质疑 |
| **执行角色** | 律师 / 法务专员（P1 关键用户） |
| **持续时长** | 无固定时长，SLA 截止 30 分钟触发催办 |
| **后续状态** | `completed`（审核完成）/ `human_review_failed`（异常失败）/ `rejected`（拒绝） |

### 6.2 展示规范

#### 审核人视角（律师 / 法务专员）

**双视图主界面**（依据 analysis-summary § 四.2 Adobe + Kira 最佳实践）：

| 区域 | 内容 | 规范 |
|------|------|------|
| 左侧面板（60% 宽） | 原始 PDF 原文展示 | PDF.js 渲染，带高亮标注层，段落级精度（MVP） |
| 右侧面板（40% 宽） | AI 审核结论列表 | 按风险等级排序，含置信度标注、推理说明 |
| 联动高亮 | 点击右侧风险条目 | 左侧 PDF 自动滚动到对应段落，该段落高亮显示（蓝色边框） |

**风险条目展示**（依据 analysis-summary § 四.1 四层递进架构）：

```
每条风险项显示：
  ├── 风险等级标签（色块：Critical深红 / High橙红 / Medium橙 / Low黄）
  ├── 风险类型（如：违约责任条款）
  ├── AI 审核结论摘要（1-2 句）
  ├── 置信度展示（百分比 + 置信度类别：事实提取/条款检查/法律解释）
  ├── 法规来源引用（折叠展开，如：《合同法》第52条第3款）
  └── 操作按钮：[确认 AI 结论] [驳回 AI 结论] [添加批注]
```

**SLA 倒计时**：

- 剩余时间 > 10 分钟：不显示（避免干扰）
- 剩余时间 5–10 分钟：黄色倒计时提示（"审核将在 X 分钟后超期"）
- 剩余时间 < 5 分钟：橙色醒目提示
- 超期：红色警告 + 后台自动催办通知（problem_modeling § 三.2 场景 C）

#### 提交人 / 观察者视角（企业法务专员 P0 用户）

此时提交人处于等待状态，展示内容：
- 任务状态标签："人工审核中"（蓝色标签）
- 分配审核人信息（脱敏展示，如"律师张 XX"）
- SLA 截止时间展示（不展示倒计时，仅展示绝对时间）
- 禁止查看 AI 审核结论详情（避免影响人工审核独立性）

### 6.3 用户可执行操作

#### 审核人（律师 / 法务专员）

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 逐条确认 AI 结论 | ✅ 允许 | 每条风险项独立确认，支持部分确认 |
| 逐条驳回 AI 结论并说明 | ✅ 允许 | 必须填写驳回原因（不得为空），长度 ≥ 10 字 |
| 添加批注 | ✅ 允许 | 关联到具体风险条目或文档段落 |
| 新增人工发现的风险项 | ✅ 允许（P1 功能） | MVP 版本若支持，须标注"人工新增"区别于 AI 识别 |
| 提交审核完成 | ✅ 允许 | 所有 Critical/High 风险项必须有明确操作才可提交 |
| 拒绝审核任务 | ✅ 允许 | 流转至 `rejected`，须填写拒绝原因 |
| 查看文档元数据 | ✅ 允许 | 合同主体、金额、日期等结构化字段 |

#### 提交人（等待方）

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 撤回任务（取消人工审核） | ✅ 允许 | 流转至 `rejected`，须二次确认 |
| 催办审核人 | ✅ 允许（限每 10 分钟一次） | 发送系统通知给审核人 |
| 查看 AI 审核结论 | ❌ 禁止 | 等待人工审核结果期间不展示 AI 结论 |
| 修改文档 | ❌ 禁止 | 审核中不得更换文档 |

### 6.4 出口条件

| 出口 | 触发条件 | 跳转目标 |
|------|---------|---------|
| 审核完成 | 审核人完成所有必要操作后点击"提交" | → `completed` |
| 系统异常 | 审核过程中系统发生不可恢复错误 | → `human_review_failed` |
| 审核人拒绝 | 审核人点击"拒绝任务"并填写原因 | → `rejected` |
| 提交人撤回 | 提交人取消任务 | → `rejected` |

### 6.5 向下一状态的数据传递内容

流转至 `completed` 时，需携带：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 审核任务标识 |
| `reviewer_id` | UUID | 执行人工审核的审核人 ID |
| `review_started_at` | timestamp | 人工审核开始时间 |
| `review_completed_at` | timestamp | 人工审核完成时间 |
| `human_review_records` | array | 逐条操作记录（确认/驳回/批注，含操作时间） |
| `confirmed_risk_items` | array | 确认的风险条目 ID 列表 |
| `rejected_risk_items` | array | 驳回的风险条目 ID 列表（含驳回原因） |
| `added_risk_items` | array | 人工新增的风险条目（若有） |
| `annotations` | array | 批注内容列表（含关联位置信息） |
| `final_risk_score` | int (0-100) | 人工审核后的最终风险评分（可与 AI 评分不同） |
| `sla_met` | boolean | 是否在 SLA 时限内完成 |

---

## 七、human_review_failed — 人工审核异常失败

### 7.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | `human_reviewing` |
| **触发条件** | 人工审核进行中系统发生不可恢复错误（如：审核界面数据丢失、会话超时导致操作记录无法保存、数据库写入失败等） |
| **注意区分** | 此状态是**系统异常**，不同于审核人主动拒绝（后者流转至 `rejected`） |
| **后续状态** | `human_reviewing`（恢复继续）/ `rejected`（放弃） |

### 7.2 展示规范

- **错误描述**：说明系统异常而非审核人操作问题（避免误导用户责怪审核人）
- **操作保留提示**：明确说明已完成的操作是否已保存（"您在 14:32 之前的操作已自动保存"）
- **分配信息保留**：展示原审核人信息，避免重分配时重复操作

| 错误类型 | 用户可见文案 |
|---------|------------|
| 会话超时 / 数据未保存 | "审核会话已超时，部分操作可能未保存，请重新进入审核" |
| 系统内部错误 | "审核系统遇到异常，已自动记录，请重新开始审核" |
| 网络中断导致操作丢失 | "网络中断，审核操作未能提交，请检查网络后重试" |

### 7.3 用户可执行操作

| 操作 | 是否允许 | 角色 | 说明 |
|------|---------|------|------|
| 重新进入审核 | ✅ 允许 | 审核人 | 恢复已保存的操作状态，流转回 `human_reviewing` |
| 重新分配审核人 | ✅ 允许 | 管理员 | 将任务重新分配给其他审核人 |
| 放弃此次审核 | ✅ 允许 | 提交人 / 管理员 | 流转至 `rejected` |
| 查看已保存的操作记录 | ✅ 允许 | 审核人 / 管理员 | 审计日志查看，支持还原操作进度 |

### 7.4 出口条件

| 出口 | 触发条件 | 跳转目标 |
|------|---------|---------|
| 恢复审核 | 审核人或管理员点击"重新进入审核" | → `human_reviewing`（恢复已保存进度） |
| 放弃 | 提交人或管理员点击"放弃任务" | → `rejected` |

### 7.5 向下一状态的数据传递内容

流转至 `human_reviewing`（恢复）时，需携带：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 保持同一任务标识 |
| `reviewer_id` | UUID | 恢复时的审核人 ID（可能与原审核人不同） |
| `saved_progress` | object | 最后一次成功保存的操作状态快照 |
| `failure_timestamp` | timestamp | 失败发生时间（审计用途） |
| `failure_reason` | string | 内部失败原因（不展示给用户） |
| `retry_count` | int | 已恢复次数（限制最大恢复次数防止无限循环） |

## 八、rejected — 审核任务被拒绝

### 8.1 状态定义

| 属性 | 内容 |
|------|------|
| **前置状态** | 任意非终态状态（uploaded / parsing / parsed / auto_reviewing / auto_reviewed / human_reviewing 均可流转至此） |
| **触发条件** | 用户主动取消 / 审核人主动拒绝 / 系统判断文档不符合处理条件 |
| **终态标志** | `rejected` 是**终态**，不可恢复，不可回退 |
| **后续状态** | 无（用户需创建新任务重新提交） |

### 8.2 展示规范

#### 拒绝原因分类

| 来源 | 原因类型 | 用户可见说明 |
|------|---------|------------|
| 用户主动取消 | 用户操作 | "您已取消此审核任务" |
| 审核人拒绝 | 人工操作 | "审核人已拒绝此任务：[填写的拒绝原因]" |
| 系统超时拒绝 | 系统行为 | "任务因长时间未处理已被系统关闭" |
| 文档违规 | 系统判断 | "文档内容不符合审核条件：[具体原因]" |

#### 视觉规范

- **状态图标**：红色 X 图标（终态不可恢复的明确信号）
- **拒绝原因**：必须展示，禁止隐藏
- **任务时间线**：展示从上传到被拒绝的完整时间线（审计追溯）
- **行动引导**：展示"重新上传文档"按钮（跳转至新任务创建），而非提供"恢复"幻觉

#### 历史记录保留

- `rejected` 任务在任务列表中保留（标注"已拒绝"灰色标签），支持用户查阅历史
- 不自动删除，保留至用户主动归档或系统定期清理

### 8.3 用户可执行操作

| 操作 | 是否允许 | 说明 |
|------|---------|------|
| 查看拒绝原因 | ✅ 允许 | 必须展示，且拒绝原因需来自拒绝记录不可伪造 |
| 重新上传文档（创建新任务） | ✅ 允许 | 基于同一文档重新发起，创建全新任务 |
| 下载原始文档 | ✅ 允许 | 用户可取回文件 |
| 查看已完成阶段的审计记录 | ✅ 允许 | 查看被拒绝前已完成步骤的操作日志（审计用途） |
| 恢复 / 回退此任务 | ❌ 禁止 | 终态不可逆（problem_modeling § 四.4.3 状态流转单向性） |
| 修改拒绝原因 | ❌ 禁止 | 拒绝原因一旦记录不得修改（审计合规要求） |

### 8.4 出口条件

`rejected` 为终态，无出口流转。用户的后续操作均创建全新任务。

### 8.5 向下一状态的数据传递内容

`rejected` 为终态，不向任何后续状态传递数据。但终态写入时，需完整记录以下字段供审计：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `review_task_id` | UUID | 审核任务标识 |
| `rejected_at` | timestamp | 拒绝时间 |
| `rejected_by` | string | 拒绝来源（user / reviewer / system） |
| `rejected_by_user_id` | UUID | 操作用户 ID（system 操作时为空） |
| `rejection_reason` | string | 拒绝原因（必填，不可为空） |
| `previous_status` | string | 被拒绝前的上一个状态（用于审计追溯） |
| `completed_steps` | array | 已完成阶段列表（审计完整性） |

## 九、状态交互汇总对比

| 状态 | 用户等待时感知 | 最关键操作 | 必须展示的信息 | 终态 |
|------|-------------|----------|--------------|------|
| `auto_reviewing` | 进度明确、分层展示 | 取消任务 | 三层进度 + 预计时间 | 否 |
| `auto_review_failed` | 失败原因清晰 | 重试 / 转人工 | 失败原因 + 恢复操作 | 否 |
| `parse_failed` | 失败原因清晰 | 重新上传 | 失败原因 + 格式要求 | 是 |
| `human_reviewing` | 等待中（审核人视角：工作台） | 逐条确认/驳回 + 提交 | SLA 倒计时 + AI 结论 + 双视图 | 否 |
| `human_review_failed` | 异常中断 | 恢复审核 | 已保存进度说明 + 失败原因 | 否 |
| `rejected` | 任务终止 | 重新上传 | 拒绝原因 + 时间线 | 是 |
