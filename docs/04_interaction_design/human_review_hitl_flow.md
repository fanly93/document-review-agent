## 一、文档目标与设计范围

### 1.1 文档目标

本文档聚焦于 **HITL（Human-in-the-Loop，人机交互循环）** 节点的完整交互链路设计，定义人工审核者在进入审核界面后的所有操作路径、界面规范与状态影响，为后续前端实现（09_frontend_plan）和数据模型设计（07_data_model）提供直接输入。

### 1.2 设计范围

本文档覆盖以下四个维度：

| 维度 | 内容 |
|------|------|
| **任务管理层** | HITL 触发条件、任务分配与状态流转规则 |
| **审核界面层** | 双视图布局规范、信息架构与展示规则 |
| **交互操作层** | 同意、编辑、驳回、批注四类操作的完整定义与数据字段 |
| **完成流程层** | 审核结束后的状态流转与审计记录规范 |

### 1.3 设计边界（不在本文档范围内）

- 自动审核（Auto Review）阶段的内部逻辑（属于系统架构设计范畴）
- 文档上传与解析流程（属于场景 A 交互设计范畴）
- 审核结果报告的只读查阅界面（属于场景 D 交互设计范畴）
- 多角色团队协作批注（MVP 范围外，定位为 V2 功能）

## 二、HITL 触发条件与任务管理层

### 2.1 HITL 触发条件

依据 `problem_modeling.md § 三 · 场景 C`，以下任意一条满足时，系统自动将 ReviewTask 状态从 `auto_reviewed` 流转至 `human_reviewing`，进入 HITL 节点：

| 触发条件 | 触发类型 | 说明 |
|---------|---------|------|
| 风险等级 = Critical 或 High | 系统强制触发 | 不可由用户或前端绕过 |
| AI 置信度 < 50% | 系统强制触发 | 适用于任意单条 RiskItem |
| 文档类型未能识别 | 系统强制触发 | 进入 Layer 1 分类失败时触发 |
| 用户手动标记"质疑此结论" | 用户主动触发 | 用户在审核结果查阅界面的操作 |

**强制规范**：HITL 触发判断必须在后端执行，前端无权判断是否触发或绕过触发。

### 2.2 任务状态机（HITL 相关节点）

以下为完整 10 态状态机中与 HITL 相关的节点及其流转规则：

```
auto_reviewed
     │
     ├── [无触发条件] ──────────────── completed（终态）
     │
     └── [满足任意触发条件]
              │
         human_reviewing ─── [超时/系统异常] ─── human_review_failed
              │
         [审核人完成操作]
              │
         completed（终态）
              │
         [审核人执行整体任务驳回]
              │
         rejected（终态，含驳回原因）
```

**状态流转规则**：
1. 状态只能按上述路径单向流转，`completed` 和 `rejected` 均为终态，不可回退
2. `human_reviewing` → `completed` 的流转，要求审核人对所有 Critical/High 风险项执行过明确操作（同意、编辑或单条驳回），不得直接跳过
3. `human_review_failed` 为系统异常态，需人工介入处理，不计入正常操作路径

### 2.3 任务分配规则

| 规则 | 描述 |
|------|------|
| 分配对象 | 律师 / 法务专员（用户角色 P1/P0，来源：`problem_modeling.md § 二`） |
| 分配方式（MVP） | 系统按任务队列顺序分配，不支持自动优先级排序（V2 功能） |
| 通知机制 | 任务分配后立即发送通知（站内通知，MVP 不要求邮件/IM 集成） |
| 超时催办 | 任务进入 `human_reviewing` 状态超过 30 分钟未完成 → 触发催办通知 |
| SLA 监控 | 每 5 分钟检查一次超期任务（来源：`analysis-summary.md § 三·3.4`） |

## 三、审核界面层：双视图布局规范

### 3.1 整体布局结构

审核界面采用**左右双视图（Split View）**布局，这是竞品分析确认的最优模式（来源：`analysis-summary.md § 四·4.2`，Adobe Acrobat AI + Kira Systems 共同验证）：

```
┌─────────────────────────────────────────────────────────────┐
│  顶部工具栏：任务信息 | 进度指示 | 提交/驳回整体任务 按钮         │
├───────────────────────────┬─────────────────────────────────┤
│                           │                                 │
│   左视图：原文 PDF         │   右视图：AI 审核结论面板          │
│   （带高亮标注层）          │   （风险列表 + 操作区）            │
│                           │                                 │
│   - PDF 原文渲染            │   - 风险项列表（可滚动）           │
│   - 高亮覆盖层（颜色分级）   │   - 当前焦点风险项详情            │
│   - 页码导航               │   - 操作按钮区（同意/编辑/驳回）   │
│                           │   - 批注输入区                   │
│                           │                                 │
└───────────────────────────┴─────────────────────────────────┘
│  底部状态栏：已处理 N / 共 M 条 | 剩余 Critical X 条            │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 左视图：原文 PDF 规范

**功能要求**：

| 功能 | 规范 | 依据 |
|------|------|------|
| PDF 渲染 | 使用 PDF.js 渲染原文 | `analysis-summary.md § 五·5.3` |
| 高亮覆盖层 | 在 PDF.js 渲染层之上叠加自定义高亮层 | 同上 |
| 高亮颜色分级 | 与风险等级色彩规范一致（见下节） | `analysis-summary.md § 四·4.1` |
| 定位精度（MVP） | 段落级定位（`RiskItem.location_paragraph`） | `problem_modeling.md § 三·场景C` |
| 多高亮共存 | 同一段落可存在多个高亮标注，颜色取最高风险等级 | 设计决策 |
| 定位触发 | 点击右视图任意风险条目 → 左侧自动滚动到对应位置并高亮 | Adobe+Kira 最佳实践 |
| 页码导航 | 支持页码跳转，不影响当前高亮状态 | 基础需求 |

**高亮颜色规范**（与自动审核结果展示保持一致）：

| 风险等级 | 颜色 | 色值 |
|---------|------|------|
| Critical（8-10 分） | 深红 | `#DC143C` |
| High（6-7 分） | 橙红 | `#FF4500` |
| Medium（4-5 分） | 橙色 | `#FFA500` |
| Low（2-3 分） | 黄色 | `#FFE000` |
| Minimal（0-1 分） | 绿色 | `#90EE90` |

### 3.3 右视图：AI 审核结论面板规范

**信息架构（4 层递进，来源：`analysis-summary.md § 四·4.1`）**：

```
Level 1：任务整体风险评分（0-100 色阶仪表盘）
Level 2：分类风险统计（条款类型 × 风险等级数量）
Level 3：风险项列表（可按风险等级/条款类型排序过滤）
Level 4：单条风险项详情展开区（焦点态）
```

**单条风险项（Level 4）必须展示的信息**：

| 字段 | 来源 | 说明 |
|------|------|------|
| 风险类型标签 | `RiskItem.risk_type` | 如：条款缺失、不对称责任、模糊表述等 |
| 风险等级 | `RiskItem.risk_level` | Critical/High/Medium/Low/Minimal |
| 置信度 + 置信度类别 | `RiskItem.confidence` + `confidence_category` | 事实提取/条款检查/风险评估，颜色区分 |
| 原文摘录 | `RiskItem.source_text` | 对应高亮位置的原文文字 |
| AI 推理说明 | `RiskItem.reasoning` | 自然语言说明，如"该条款被标记为高风险，因为：1.…" |
| 法规来源引用 | `SourceReference[]` | 折叠展开，显示完整引用路径 |
| 操作状态 | `HumanReview.action` | 当前该条目已被执行的操作（待处理/已同意/已编辑/已驳回） |

**处理进度指示**：
- 顶部工具栏实时显示"已处理 N / 共 M 条"
- 底部状态栏显示"剩余 Critical X 条 | High Y 条"
- 所有 Critical/High 条目未处理时，"完成审核"按钮置灰不可点击

## 四、交互操作层：四类操作的完整定义

### 4.1 操作概览

审核人在 HITL 界面可对每条风险项执行以下四类操作：

| 操作 | 英文标识 | 适用对象 | 对 ReviewTask 状态的影响 |
|------|---------|---------|------------------------|
| **同意** | `approve` | 单条 RiskItem | 不影响，仅记录操作 |
| **编辑** | `edit` | 单条 RiskItem | 不影响，记录操作与修改内容 |
| **单条驳回** | `reject_item` | 单条 RiskItem | 不影响，将该条 RiskItem 标记为"审核人驳回" |
| **整体任务驳回** | `reject_task` | 整个 ReviewTask | 将 ReviewTask 状态流转至 `rejected`（终态） |

此外，审核人可在任意风险项上添加**批注**（`annotate`），批注不单独构成对该风险项的最终操作判断，而是作为辅助说明附加在同意/编辑/驳回操作上。

### 4.2 同意操作（approve）

**定义**：审核人认同该条 AI 风险识别结论，不需要修改，标记为已确认。

**触发方式**：点击风险项操作区的"同意"按钮。

**操作效果**：
- 该风险项在右视图列表中显示"已同意"状态标签（绿色）
- 左视图对应高亮区域保留，但高亮边框加粗表示已处理
- 写入 `HumanReview` 记录（见数据字段规范 § 4.6）

**约束**：同意操作后仍可撤销（在当前会话内），撤销后恢复为"待处理"状态。

### 4.3 编辑操作（edit）

**定义**：审核人对 AI 的风险识别结论存在异议，需要修改结论内容（如修正风险描述、调整风险等级、更新推理说明），但认为该风险点确实存在，不予驳回。

**触发方式**：点击风险项操作区的"编辑"按钮，进入编辑模式。

**可编辑字段**（MVP 范围内）：

| 可编辑字段 | 字段标识 | 说明 |
|-----------|---------|------|
| 风险等级 | `risk_level` | 下拉选择，5 个等级 |
| 风险描述 | `risk_description` | 富文本/纯文本编辑，支持多行 |
| AI 推理说明 | `reasoning` | 纯文本编辑，审核人可补充或修正 |

**不可编辑字段**（保留 AI 原始记录）：
- 风险类型（`risk_type`）
- 原文定位（`location_page`、`location_paragraph`）
- 置信度及置信度类别（AI 计算值，不可人工修改）

**编辑记录数据字段**（为 HumanReview 数据模型提供直接输入）：

每次编辑操作必须完整记录以下字段，用于合规审计追溯：

| 字段 | 字段标识 | 类型 | 说明 |
|------|---------|------|------|
| 操作人 ID | `operator_id` | String（UUID） | 执行编辑的用户唯一标识，对应 `User.id` |
| 操作时间戳 | `operated_at` | DateTime（ISO 8601，UTC） | 编辑操作提交的精确时间，精确到毫秒 |
| 编辑字段名 | `edited_field` | String | 被修改的字段名称（如 `risk_level`、`risk_description`） |
| 原值 | `original_value` | String | 修改前的字段值（字符串化存储，枚举值取其字符串名） |
| 新值 | `new_value` | String | 修改后的字段值（同上） |

**设计说明**：上述五个字段是后续 `HumanReview` 数据实体中编辑记录的直接输入，数据模型设计阶段（07_data_model）必须确保这五个字段均有对应的数据库列，且原值/新值不可为 NULL（编辑操作产生前后值都必须存在）。

**操作效果**：
- 该风险项在右视图显示"已编辑"状态标签（蓝色），并展示编辑内容 diff
- 写入 `HumanReview` 记录（action = `edit`，包含完整编辑字段列表）
- 左视图对应高亮颜色按修改后的风险等级更新

### 4.4 驳回操作（reject）

驳回操作分为两个层级，其作用对象和对 ReviewTask 最终状态的影响完全不同：

#### 4.4.1 单条驳回（reject_item）

**定义**：审核人认为该条 AI 风险识别结论有误（误报），在本文档中该风险点实际不存在，予以否定。

**触发方式**：点击风险项操作区的"驳回"按钮，进入驳回模式，必须填写驳回理由后方可提交。

**操作效果**：
- 该条 `RiskItem` 被标记为 `reviewer_rejected`，在最终报告中不作为有效风险项计入
- 整体风险评分在最终 `completed` 状态报告中，需剔除已驳回的 RiskItem 重新计算（或标注"人工复核后评分"）
- 该风险项在右视图显示"已驳回"状态标签（灰色），可展开查看驳回理由
- 左视图对应高亮消除（或改为灰色虚线框表示已驳回）
- `ReviewTask` 状态**不发生变化**，仍保持 `human_reviewing`，任务继续等待其他条目处理完毕

**对 ReviewTask 状态的影响**：无。单条驳回只影响对应 RiskItem 的有效性标记，不影响整体任务流转。

**必须填写字段**：
- 驳回理由（`reject_reason`）：纯文本，必填，最少 10 字符（防止空泛驳回影响审计质量）

#### 4.4.2 整体任务驳回（reject_task）

**定义**：审核人认为本次 AI 审核输出质量严重不可靠，或文档存在根本性问题（如文档版本错误、文档内容不完整），需要将整个审核任务撤销，要求重新提交。

**触发方式**：点击顶部工具栏的"驳回整体任务"按钮，弹出二次确认对话框，必须填写整体驳回原因。

**操作效果**：
- `ReviewTask` 状态立即流转至 `rejected`（终态）
- 所有子 `RiskItem` 的状态不再更新（随任务进入终态，不单独标记）
- 系统生成一条任务级 `HumanReview` 记录（action = `reject_task`，包含整体驳回原因）
- 任务发起人（文档上传者）收到通知，包含驳回原因
- `rejected` 为终态，**不可恢复**，如需重新审核必须重新上传文档并创建新的 ReviewTask

**对 ReviewTask 状态的影响**：立即且不可逆地将 ReviewTask 状态更新为 `rejected`（终态）。这与单条驳回形成本质区别：单条驳回是条目级别的意见标记，整体任务驳回是任务级别的终止决策。

**必须填写字段**：
- 整体驳回原因（`task_reject_reason`）：纯文本，必填，最少 20 字符

**二次确认提示文案**：
> "您即将驳回本审核任务。驳回后，任务将终止且不可恢复。文档发起人将收到通知，需重新提交文档进行审核。确认继续？"

#### 4.4.3 两类驳回的对比总结

| 对比维度 | 单条驳回（reject_item） | 整体任务驳回（reject_task） |
|---------|----------------------|--------------------------|
| 作用对象 | 单条 RiskItem | 整个 ReviewTask |
| 触发位置 | 风险项操作区（右视图） | 顶部工具栏 |
| 必填原因 | 是（最少 10 字符） | 是（最少 20 字符） |
| 二次确认 | 否 | 是（弹窗确认） |
| ReviewTask 状态变化 | 无 | `human_reviewing` → `rejected`（终态） |
| RiskItem 状态变化 | 当前条目标记为 `reviewer_rejected` | 所有条目随任务终止，不单独更新 |
| 是否可恢复 | 可撤销（当前会话内） | 不可恢复 |
| 后续操作 | 任务继续，处理其他条目 | 任务终止，需重新提交文档 |

### 4.5 批注操作（annotate）

**定义**：审核人对某条风险项添加说明性文字，作为辅助信息附在该条目上，不单独构成最终操作判断。批注通常与同意/编辑/驳回组合使用。

**触发方式**：点击风险项的"添加批注"图标，展开批注输入框。

**操作规则**：
- 批注内容可独立存储（即使未执行同意/编辑/驳回，批注也可保存）
- 批注不影响该风险项的处理状态（"待处理"状态的条目即使有批注，仍不视为"已处理"）
- 批注内容写入 `HumanReview` 记录（action = `annotate`）
- MVP 阶段批注为单人单次，不支持多人批注线程（V2 功能）

### 4.6 操作记录数据字段汇总（HumanReview 实体输入）

以下为所有 HITL 操作在 `HumanReview` 数据实体中的字段定义，直接为 `07_data_model` 阶段提供输入：

| 字段 | 字段标识 | 类型 | 说明 | 是否必填 |
|------|---------|------|------|---------|
| 记录 ID | `id` | UUID | 主键，系统自动生成 | 系统生成 |
| 所属任务 ID | `review_task_id` | UUID | 关联 ReviewTask | 必填 |
| 所属风险项 ID | `risk_item_id` | UUID / NULL | 操作对象（任务级操作时为 NULL） | 条件必填 |
| 操作人 ID | `operator_id` | UUID | 执行操作的用户 ID | 必填 |
| 操作类型 | `action` | Enum | `approve` / `edit` / `reject_item` / `reject_task` / `annotate` | 必填 |
| 操作时间戳 | `operated_at` | DateTime（UTC） | 操作提交的精确时间，精确到毫秒 | 必填 |
| 批注/原因内容 | `comment` | Text / NULL | 批注内容、驳回原因等 | 条件必填 |
| 编辑字段名 | `edited_field` | String / NULL | 仅 edit 操作时填写 | 条件必填 |
| 原值 | `original_value` | String / NULL | 仅 edit 操作时填写 | 条件必填 |
| 新值 | `new_value` | String / NULL | 仅 edit 操作时填写 | 条件必填 |

## 五、完成流程层：审核结束与状态流转

### 5.1 正常完成路径（complete）

**前提条件**：
- 所有 Critical 和 High 风险项已执行过至少一个操作（同意、编辑或单条驳回）
- 未执行整体任务驳回

**触发方式**：点击顶部工具栏的"完成审核"按钮（仅在前提条件满足时可点击）。

**系统执行动作**：
1. 将 `ReviewTask.status` 从 `human_reviewing` 更新为 `completed`
2. 记录 `ReviewTask.completed_at` 时间戳
3. 汇总本次 HITL 操作记录（所有 `HumanReview` 条目）
4. 重新计算最终风险评分（剔除 `reviewer_rejected` 的 RiskItem）
5. 生成最终审核报告（供场景 D 查阅）
6. 通知文档发起人"审核已完成"

### 5.2 整体任务驳回路径（reject）

详见 § 4.4.2，此处不重复描述。

### 5.3 完成后的审计记录规范

依据 `problem_modeling.md § 四·4.3`（关键业务规则第 4 条），**所有状态变更和人工操作必须写入不可变审计日志**，以支持合规审计追溯。

HITL 阶段结束后，以下信息必须完整保留：

| 审计项 | 记录位置 | 不可变要求 |
|--------|---------|----------|
| 任务状态流转记录 | `ReviewTask.status_history`（或独立 AuditLog 表） | 只追加，不修改 |
| 每条 HumanReview 操作记录 | `HumanReview` 表 | 写入后不可删除/修改 |
| 编辑操作的原值与新值 | `HumanReview.original_value` / `new_value` | 必须保留完整前后值 |
| 审核完成时间 | `ReviewTask.completed_at` | 不可修改 |
| 向量库版本（来源自 auto_review 阶段） | `ReviewTask.vector_db_version` | 与审核报告绑定，支持事后重现 |

## 六、关键设计约束与原则

### 6.1 源自业务调研的强制约束

以下约束直接来自上游文档，在 HITL 界面设计中不可违背：

| 约束 | 来源文档 | 具体要求 |
|------|---------|---------|
| HITL 触发是系统行为 | `problem_modeling.md § 四·4.3` | 后端判断触发，前端无权绕过 |
| 审计日志全量记录 | `problem_modeling.md § 四·4.3` | 所有操作写入不可变日志 |
| 置信度分级强绑定展示 | `problem_modeling.md § 四·4.3` | UI 颜色和警示等级由后端字段决定 |
| HITL 质量监控 | `business-summary.md § 四·4.2` | 实时监控审核确认时长，中位数过低时触发质量审查 |
| 状态流转单向性 | `problem_modeling.md § 四·4.3` | completed/rejected 无法回退 |

### 6.2 竞品分析确认的最佳实践

| 设计决策 | 竞品来源 | 本系统采纳方式 |
|---------|---------|--------------|
| 左 PDF + 右结论双视图 | Adobe Acrobat AI + Kira Systems | 完全采纳，作为 P0 核心交互 |
| 段落级原文定位（MVP） | Kira Systems | MVP 实现段落级，V1 升级句子级 |
| 4 层递进风险信息架构 | Kira Systems 金字塔模型 | 在右视图面板中实现 |
| 来源引用折叠展开 | LexisNexis + Perplexity | 法规来源默认折叠，点击展开 |
| AI 推理自然语言说明 | Luminance Decipher XAI | P1 功能，MVP 必须实现 |

### 6.3 HITL 质量监控机制

为避免"警觉疲劳（Alert Fatigue）"导致审核人习惯性点击确认（来源：`business-summary.md § 三`），系统需要以下质量保障机制：

| 监控指标 | 阈值 | 触发动作 |
|---------|------|---------|
| 单次审核完成总时长 | < 3 分钟（异常快） | 后台质量告警，记录至质量监控日志 |
| 所有风险项操作均为"同意" | 100% approve 率 | 后台质量告警（可能存在习惯性确认） |
| 连续 N 次全部同意 | N ≥ 5 次任务 | 触发人工质量抽查 |
| Critical 条目单条审核时长 | < 30 秒 | 后台质量告警 |
