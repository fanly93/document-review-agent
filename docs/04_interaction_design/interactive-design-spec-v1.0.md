## 一、全链路交互总览

### 1.1 完整审核链路全景图

```
用户上传文档
     ↓
[上传交互层]
  文件校验（格式/大小）
  文档类型自动识别
  分片上传 + WebSocket 实时进度
     ↓
  status: uploaded
     ↓
[解析层]
  多引擎策略：直接文本提取 → OCR → 降级人工
  OCR 质量门控（质量分 < 70% 转人工，≥ 85% 通过）
     ↓
  status: parsing
     ↓────────────────────────────── parse_failed（终态）
     ↓（解析成功）
  status: parsed
     ↓
[自动审核层]
  Layer 1：格式校验 & 文档分类（< 1s）
  Layer 2：条款识别 & 规则匹配（3-10s）
  Layer 3：LLM 深度分析（10-30s）
     ↓
  status: auto_reviewing
     ↓────────────────────────────── auto_review_failed
     ↓（审核完成）
  status: auto_reviewed
     ↓
  ┌──────────────────────────────┐
  │ HITL 触发判断（系统行为）       │
  │ · 风险等级 Critical/High      │
  │ · 置信度 < 50%               │
  │ · 未知文档类型                │
  │ · 用户手动标记质疑             │
  └──────────────────────────────┘
     ↓（触发HITL）                    ↓（无需HITL）
[人工审批层]                        status: completed（终态）
  任务分配 + 接收通知
  双视图审核界面（左PDF / 右结论）
  操作：同意 / 编辑 / 驳回 / 批注
     ↓
  status: human_reviewing
     ↓────────────────────────────── human_review_failed
     ↓────────────────────────────── rejected（终态）
     ↓（审核完成）
  status: completed（终态）

任意状态 ─────────────────────────── rejected（终态，含原因）
```

### 1.2 链路分工与衔接边界

| 链路段 | 负责文档 | 入口状态 | 出口状态 |
|--------|---------|---------|---------|
| 上传 → 解析 → 保存 | `upload_parse_save_flow.md` | 无（初始） | `parsed` / `parse_failed` |
| 自动审核中间状态 | `review_state_handling_flow.md` | `parsed` | `auto_reviewed` / 各失败态 / `rejected` |
| 人工审批（HITL） | `human_review_hitl_flow.md` | `human_reviewing` | `completed` / `rejected` |

## 二、上传 → 解析 → 保存链路规范摘要

> 完整规范见：`upload_parse_save_flow.md`

### 2.1 支持格式与上传方式（MVP）

- **支持格式**：PDF（`.pdf`）、Word（`.docx`、`.doc`）
- **上传方式**：拖拽上传 + 点击选择，支持批量（最多 10 个/次）
- **大文件策略**：≥ 20MB 自动启用分片上传（5MB/片，最多 3 并发），断点续传有效期 24 小时
- **不支持类型**：明确拦截融资股权文件；`.txt`、`.xlsx` 等格式提示"不支持"而非报错

### 2.2 多引擎解析策略

```
Step 1：直接文本提取（PyMuPDF/pdfplumber）
   → 成功且无乱码 → 进入质量门控
   → 失败 / 乱码率 > 15% → 升级 OCR

Step 2：OCR 引擎（PaddleOCR，中文合同优先）
   → OCR 质量分 ≥ 85%：通过，进入审核流程
   → 质量分 70-84%：标记"解析质量中等"，带警示继续
   → 质量分 < 70%：转人工降级，触发 parse_failed 或人工辅助

Step 3：降级人工（质量门控不通过时）
   → 系统提示用户"扫描质量不足，建议重新扫描"
   → 提供人工辅助通道
```

### 2.3 WebSocket 实时进度规范

| 阶段 | 推送事件 | 进度范围 | 展示文案 |
|------|---------|---------|---------|
| 上传中 | `upload_progress` | 0–40% | "上传中 (XX%)" |
| OCR/文本提取 | `parse_progress` | 40–70% | "文档解析中…" |
| 质量门控 | `quality_check` | 70–85% | "质量检测中…" |
| 完成 | `parse_complete` | 100% | "解析完成，进入审核" |
| 失败 | `parse_failed` | — | 对应错误文案 |

### 2.4 上传阶段错误类型与处理

| 错误类型 | 用户提示 | 处理路径 |
|---------|---------|---------|
| 格式不支持 | "不支持此文件格式，请上传 PDF 或 Word" | 前端拦截，不提交 |
| 文件超大（>50MB） | "文件大小超过 50MB，请压缩后重新上传" | 前端拦截 |
| 文件损坏 | "文件可能已损坏，无法读取，请重新上传" | 后端检测，进入 parse_failed |
| OCR 质量不足 | "扫描质量不足（质量分 XX%），建议重新扫描后上传" | 进入降级路径 |
| 不支持文档类型 | "融资股权文件不在当前 AI 审核范围内，请联系法务人工处理" | 硬拦截，不进入审核流程 |

## 三、自动审核中间状态链路规范摘要

> 完整规范见：`review_state_handling_flow.md`

### 3.1 auto_reviewing — 三层进度展示规范

| 层级 | 展示文案 | 进度权重 | 预计时长 |
|------|---------|---------|---------|
| Layer 1 | "格式校验 & 文档分类中…" | 5% | < 1s |
| Layer 2 | "条款识别 & 规则匹配中…" | 35% | 3–10s |
| Layer 3 | "LLM 深度分析中…" | 60% | 10–30s |

**关键约束**：
- 所有进度通过 WebSocket 推送，禁止前端轮询
- 此状态下禁止展示任何部分审核结论
- 禁止用户手动触发 HITL（HITL 触发是系统行为）

### 3.2 各失败态处理规范

| 状态 | 错误分类 | 用户操作项 | 系统行为 |
|------|---------|----------|---------|
| `auto_review_failed` | 模型超时 / 置信度极低 / 系统错误 | 手动重试 / 升级人工 | 记录失败原因，支持 1 次自动重试 |
| `parse_failed` | OCR失败 / 格式问题 / 文件损坏 | 重新上传 / 联系支持 | 终态，保留历史记录 |
| `human_review_failed` | 系统异常 / 审核人操作失败 | 重新分配 / 联系管理员 | 区分系统错误与人工操作失败，保留已保存进度 |
| `rejected` | 任意原因驳回 | 查看驳回原因 / 重新提交 | 终态，不可恢复，完整审计日志 |

### 3.3 SLA 监控与催办规则（跨状态通用）

| 监控项 | 阈值 | 触发动作 |
|-------|------|---------|
| SLA 检查周期 | 每 5 分钟 | 系统自动扫描超期任务 |
| human_reviewing 超期催办 | 30 分钟未处理 | 发送催办通知给审核人 |
| human_reviewing 严重超期 | 60 分钟未处理 | 触发任务重新分配 |
| auto_reviewing 超时 | 5 分钟无响应 | 视为 auto_review_failed，进入重试流程 |

## 四、人工审批（HITL）链路规范摘要

> 完整规范见：`human_review_hitl_flow.md`

### 4.1 双视图审核界面规范

```
┌─────────────────────────────────────────────────────────────────┐
│ 顶部工具栏：文档名 | 整体风险评分 | 进度(已处理N/M) | 完成审核 | 驳回整体任务 │
├─────────────────────────┬───────────────────────────────────────┤
│                         │                                       │
│    左侧：原文 PDF        │    右侧：AI 审核结论面板              │
│    (60% 宽度)           │    (40% 宽度)                        │
│                         │                                       │
│  · PDF.js 渲染          │  Level 1: 整体风险评分（0-100）       │
│  · 高亮标注层叠加        │  Level 2: 分类风险看板               │
│  · 段落高亮颜色          │  Level 3: 风险列表（可排序/过滤）     │
│    对应风险等级           │  Level 4: 条款详情（展开式）         │
│  · 缩放/翻页控件         │                                       │
│                         │  每条风险项操作区：                   │
│                         │  [同意] [编辑] [驳回] [批注]         │
│                         │                                       │
└─────────────────────────┴───────────────────────────────────────┘
```

**原文定位交互**：点击右侧任意风险条目 → 左侧 PDF 自动滚动至对应段落并高亮，高亮颜色与风险等级一致。

### 4.2 四类操作规范

#### 同意（approve）
- 单条确认：点击"同意"按钮，条目标记绿色"已确认"，计入已处理计数
- 批量确认：多选后一键确认，适用于低风险条目批量处理
- 完成条件：所有 Critical 和 High 风险项均已处理（同意/编辑/单条驳回）

#### 编辑（edit）
- 可编辑字段：风险等级、风险描述、AI 推理说明
- 不可编辑字段：风险类型、原文定位、置信度（保留 AI 原始记录）
- 编辑后：条目标记蓝色"已编辑"，展示原值与新值 diff
- 审计字段：操作人ID、时间戳、编辑字段名、原值、新值（5 个必填字段）

#### 驳回（reject）

| 类型 | 作用对象 | 对 ReviewTask 的影响 |
|------|---------|-------------------|
| 单条驳回（reject_item） | 单条 RiskItem | 无，任务继续 |
| 整体任务驳回（reject_task） | 整个 ReviewTask | 立即流转至 `rejected`（终态），不可恢复 |

- 单条驳回：必填驳回理由（≥ 10 字符），条目标记灰色"已驳回"
- 整体任务驳回：必填原因（≥ 20 字符）+ 二次确认弹窗

#### 批注（annotate）
- 可独立添加，不影响条目处理状态
- MVP 阶段单人单次，不支持多人线程回复（V2 功能）
- 批注不作为完成审核的前提条件

### 4.3 HITL 质量监控指标

| 监控指标 | 阈值 | 处理动作 |
|---------|------|---------|
| 单次审核总时长 | < 3 分钟 | 后台质量告警 |
| 全部条目均选择"同意" | 100% approve | 后台告警（疑似习惯性确认） |
| 连续 N 次全部同意 | N ≥ 5 次 | 触发人工质量抽查 |
| Critical 条目单条审核时长 | < 30 秒 | 后台质量告警 |

## 五、跨链路一致性规范

### 5.1 状态命名规范（所有模块统一）

| 状态值 | 语义 | 类型 |
|-------|------|------|
| `uploaded` | 已上传，等待解析 | 中间态 |
| `parsing` | 解析中 | 中间态 |
| `parsed` | 解析完成，等待自动审核 | 中间态 |
| `parse_failed` | 解析失败 | **终态** |
| `auto_reviewing` | 自动审核中 | 中间态 |
| `auto_review_failed` | 自动审核失败 | 可恢复失败态 |
| `auto_reviewed` | 自动审核完成 | 中间态 |
| `human_reviewing` | 人工审核中 | 中间态 |
| `human_review_failed` | 人工审核失败 | 可恢复失败态 |
| `completed` | 审核完成 | **终态** |
| `rejected` | 任务被驳回 | **终态** |

**规则**：
- 三个终态（`parse_failed`、`completed`、`rejected`）不可回退
- 状态流转只能沿定义路径单向推进
- 任意状态均可流转至 `rejected`（用户主动取消或整体驳回）

### 5.2 进度反馈统一规范

| 反馈类型 | 实现方式 | 适用链路 |
|---------|---------|---------|
| 实时进度推送 | WebSocket，禁止轮询 | 上传、解析、自动审核全程 |
| 错误提示 | Toast（轻量错误）+ 模态弹窗（阻断性错误） | 全链路 |
| 状态文案 | 动词进行时（"解析中…"），避免技术术语 | 全链路 |
| 超时催办 | 系统通知（站内 + 邮件） | human_reviewing 阶段 |

### 5.3 置信度展示统一规范

全链路任何位置展示置信度均须遵循以下分级（来源：`problem_modeling.md § 四.1`）：

| 置信度类别 | 范围 | 颜色 | 说明 |
|---------|------|------|------|
| 高置信（事实提取） | ≥ 90% | 绿色 | 合同主体、金额、日期等显式字段 |
| 中置信（条款检查） | 70–89% | 黄色 | 条款存在性检查 |
| 低置信（风险评估） | < 70% | 橙色 + 必须附推理说明 | 法律解释类输出 |

置信度类别由后端字段决定，前端不自行计算。

### 5.4 审计日志统一规范

以下操作必须写入不可变审计日志（只追加，不修改/删除）：

| 操作类型 | 记录内容 | 触发时机 |
|---------|---------|---------|
| 状态变更 | 任务ID、旧状态、新状态、时间戳、触发方（系统/用户） | 每次状态流转 |
| 人工操作 | 操作人ID、操作类型、时间戳、目标对象、详细内容 | HITL 每次操作 |
| 编辑记录 | 字段名、原值、新值（5 字段完整记录） | 每次编辑操作 |
| 向量库版本 | 审核任务绑定的向量库版本号 | 审核任务创建时 |

## 六、后续阶段输入汇总

### 6.1 为系统架构设计（06_system_architecture）提供的输入

| 架构要素 | 交互设计依据 |
|---------|------------|
| WebSocket 服务 | 上传/解析/审核全链路实时进度推送，禁止轮询 |
| 三层审核 Agent 架构 | Layer 1/2/3 对应独立处理节点，须支持节点级进度回调 |
| HITL 状态机引擎 | 10 状态单向流转，终态不可逆 |
| SLA 监控服务 | 每 5 分钟扫描，30/60 分钟双阈值催办 |
| 审计日志服务 | 不可变追加写入，全链路覆盖 |
| PDF 渲染服务 | 段落级精准定位（PDF.js + 高亮层），MVP 段落级，V1 升级句子级 |

### 6.2 为数据模型设计（07_data_model）提供的输入

| 实体 | 新增/调整字段 | 来源文档 |
|------|------------|---------|
| `Document` | `ocr_quality_score`、`parse_engine_used`、`chunk_upload_id` | upload_parse_save_flow |

| `RiskItem` | `confidence_category`（枚举：fact/clause/legal）、`reviewer_status`（pending/approved/edited/reviewer_rejected） | review_state + hitl_flow |
| `HumanReview` | `action`（枚举：approve/edit/reject_item/reject_task/annotate）、`edited_field`、`original_value`、`new_value`、`operated_at` | human_review_hitl_flow |

### 6.3 为 API 规范设计（08_api_spec）提供的输入

| API 功能 | 关键接口需求 |
|---------|------------|
| 获取上传预签名 URL | 支持分片上传，返回分片 ID |
| WebSocket 进度推送 | 统一事件格式：`{event, task_id, stage, progress, message}` |
| 文档解析状态查询 | 返回当前状态 + 最新解析结果 |
| HITL 操作提交 | POST 单条操作记录，后端校验操作合法性 |
| 整体任务驳回 | POST 任务级驳回，后端立即执行状态流转至 `rejected` |
| 完成审核 | POST 完成指令，后端校验所有 Critical/High 条目已处理 |
| 审计日志查询 | 分页查询，支持按任务ID/操作人/时间范围过滤 |

### 6.4 为前端实现计划（09_frontend_plan）提供的输入

| 前端模块 | 关键设计要求 |
|---------|------------|
| 上传组件 | 拖拽区 + 点击选择 + 分片进度条 + 续传提示弹窗 |
| 解析进度组件 | WebSocket 订阅 + 三阶段可视化（文字 + 进度条 + 状态图标） |
| 自动审核进度组件 | Layer 1/2/3 分层展示，各层独立状态图标 |
| 中间状态页 | 每个失败态独立页面，含错误说明 + 恢复操作入口 |
| 双视图审核界面 | PDF.js 渲染 + 高亮层 + 4 层风险信息架构面板 |
| 操作区按钮组 | 同意/编辑/单条驳回/批注，条件禁用逻辑 |
| 完成审核按钮 | 仅在所有 Critical/High 条目已处理时可点击 |

## 七、MVP 阶段设计约束汇总

以下约束来自上游文档，贯穿整个交互链路，**不可在设计实现中绕开**：

| 约束 | 来源 | 适用位置 |
|------|------|---------|
| 融资股权文件硬拦截 | business-summary Lead 仲裁 | 上传校验阶段 |
| AI 定位为辅助初审，不替代法律判断 | business-summary Lead 仲裁 | 所有 AI 输出界面的文案设计 |
| 置信度分级展示强制执行 | problem_modeling § 四.4.3 | 风险列表、条款详情、审核报告 |
| HITL 触发是系统行为，前端不可绕过 | problem_modeling § 四.4.3 | 自动审核完成后的路由逻辑 |
| 审计日志全量不可变记录 | problem_modeling § 四.4.3 | 状态变更、HITL 操作全程 |
| 向量库版本绑定审核任务 | business-summary § 四.2 | ReviewTask 创建时写入 |
| Prompt Injection 防护 | business-summary § 四.2 | 文档内容不直接嵌入 System Prompt |
| OCR 质量门控 | business-summary § 四.2 | 解析完成后，进入审核前 |
