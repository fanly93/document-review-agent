## 一、概述

本文档规范文档上传、格式校验、多引擎解析、OCR 质量门控、解析结果保存的完整交互链路。该链路是用户与系统的第一接触点，设计质量直接影响用户的系统信任度和日常操作效率。

**核心设计原则**：
- **可靠性优先于速度**：不追求极限处理速度，确保解析准确性和全程可追溯
- **实时反馈**：用户在全流程中都能看到当前进度和状态，避免"黑盒"感
- **多引擎降级**：直取→OCR→人工，确保最高程度的文档覆盖
- **质量门控**：OCR 输出必须通过质量检测才进入审核流程，防止垃圾数据污染

## 二、文档上传交互规范

### 2.1 支持的格式清单（MVP 范围）

| 格式 | 文件扩展名 | 说明 |
|------|----------|------|
| PDF | `.pdf` | 原生 PDF（含可搜索文本和扫描版） |
| Microsoft Word | `.docx`, `.doc` | Word 2007+ 格式及兼容版本 |
| （预留）图像 | `.jpg`, `.jpeg`, `.png` | 单页合同图片（后续支持） |

**MVP 优先级**：
- **P0（必须）**：PDF, DOCX（覆盖 95% 的合同来源）
- **P1（预留）**：图像格式（单页纸质合同扫描件）

**不支持的格式**：`.txt`, `.xlsx`, `.ppt`, `.html` 等（用户提交这些格式时，系统应明确提示"不支持"，而非报错）

---

### 2.2 上传方式与交互流程

#### 上传入口设计

**支持两种上传方式**（满足不同用户习惯）：

1. **拖拽上传**
   - 触发区域：整个上传区域可接收拖拽（至少 400×300px）
   - 视觉反馈：拖入时，上传区域背景色变浅、出现虚线边框，提示"松开即上传"
   - 多文件支持：允许用户同时拖拽多个文件（批量上传）
   - 拖拽验证：拖入非支持格式时，区域显示"不支持此文件格式"提示，用户松开时不触发上传

2. **点击选择**
   - 按钮文案：**"选择文件"** 或 **"+ 添加合同"**
   - 选择框配置：允许多选（支持批量上传）
   - 文件类型过滤：操作系统文件选择器应预过滤支持的格式

#### 文件选择限制

| 限制项 | 规范 | 提示文案 |
|--------|------|--------|
| 单文件大小限制 | ≤ 50MB | "文件大小超过 50MB，请压缩后重新上传" |
| 每批上传文件数 | ≤ 10 个 | "单次最多上传 10 个文件" |
| 单用户每天上传限额 | ≤ 100 个文件 | "今日上传配额已满，请明天再来" |

---

### 2.3 大文件处理策略

#### 分片上传触发条件

**分片上传启动**：当单个文件大小 **≥ 20MB** 时自动启用

| 配置项 | 参数值 |
|--------|--------|
| 分片大小 | 5MB（每个分片） |
| 最大并发分片数 | 3（同时上传 3 个分片） |
| 单分片超时 | 30 秒 |
| 分片失败重试次数 | 3 次 |

#### 断点续传机制

**续传触发条件**：上传过程中网络中断、用户手动暂停、浏览器刷新等

**续传交互**：
- 当用户再次选择同一文件时，系统检测该文件的上传进度
- 显示提示框：**"检测到未完成的上传，是否继续上传？"** 或 **"是否重新开始？"**
- 选择"继续"：系统跳过已上传分片，仅上传剩余分片
- 选择"重新开始"：清空进度，从第一个分片重新开始

**续传有效期**：
- 续传有效期为 **24 小时**
- 超过 24 小时的续传记录自动清除，用户需重新上传

---

### 2.4 上传进度展示规范

#### 进度条设计

**进度条样式**：
- 宽度：占上传区域的 80%，居中显示
- 高度：4px（细长形式，不占据空间）
- 背景色：浅灰色 `#E8E8E8`
- 进度填充色：蓝色 `#1890FF`

**进度更新频率**：每上传 512KB 更新一次（约 0.5-1 秒更新一次）

#### 上传状态文案展示

| 上传阶段 | 进度条显示 | 文案说明 | 用户感知 |
|---------|----------|--------|---------|
| 初始化 | 0% | "准备上传..." | 系统在验证文件，计算分片等 |
| 分片上传中 | 10%-90% | "上传中 (XX%)" | 显示实时进度百分比 |
| 分片上传完成 | 90%-95% | "正在处理..." | 等待服务器整合分片 |
| 等待审核分配 | 95%-100% | "已上传，等待处理..." | 系统已接收文件，等待进入审核队列 |
| 上传成功 | 100% | "上传成功" + 绿色勾号 | 维持 2 秒后自动消失或转向下一步 |

**进度条行为**：
- 分片上传中：线性进度推进
- 网络中断时：进度条暂停（不倒退），显示"连接已断开，10 秒后重试"
- 重连成功后：进度条继续推进

#### 批量上传进度展示

当用户批量上传多个文件时：

**选项 A - 整体进度展示**（推荐）
```
整体进度：3/10 个文件已上传
├─ file1.pdf  ✅ 已完成
├─ file2.pdf  ⏳ 上传中 (45%)
└─ file3.pdf  ⏸ 等待上传
```

**交互方式**：
- 用户可展开/折叠单个文件的详细进度
- 允许暂停/取消整个批次的上传
- 单个文件失败不影响其他文件

### 2.5 上传失败处理与错误提示规范

#### 错误分类与用户提示

| 错误类型 | 触发条件 | 用户提示文案 | 推荐行动 |
|---------|---------|-----------|--------|
| **格式错误** | 用户选择不支持的文件格式 | "不支持 .xlsx 格式，请上传 PDF 或 Word 文件" | 提示支持的格式清单，用户重新选择 |
| **大小超限** | 文件 > 50MB | "文件大小超过 50MB，请压缩后重新上传" | 指导用户压缩或分割文档 |
| **文件损坏** | 文件头校验失败 | "文件已损坏，无法打开，请确保文件完整后重试" | 用户下载原始文件，检查是否损坏 |
| **网络错误** | 上传中网络中断 | "网络连接已断开，将在 10 秒后自动重试" | 系统自动重试，用户无需操作；可手动重试 |
| **服务端错误** | 服务器返回 5xx | "上传失败，请稍后重试（错误代码：500）" | 用户可重试；若频繁失败，提示联系支持 |
| **病毒/恶意文件** | 安全扫描失败 | "文件可能含有恶意内容，已被拦截" | 联系管理员或上传其他文件 |
| **存储空间不足** | 服务端存储满 | "系统存储空间已满，请稍后再试" | 通知管理员；用户稍后重试 |

#### 错误提示的UI展示

**位置**：错误信息显示在上传区域下方或作为 Toast 通知

**样式规范**：
- 背景色：浅红色 `#FFF1F0`
- 边框：左侧 3px 红色边框 `#FF4D4F`
- 文案颜色：深红色 `#CF1322`
- 图标：错误图标（!）
- 可关闭：右上角提供关闭按钮（X）
- 自动消失时间：10 秒

**重试按钮**：
- 对于网络错误：显示"重试"按钮，允许用户手动触发重新上传
- 对于其他错误：不显示重试，用户需重新选择文件

#### 上传中断的处理

**用户手动取消上传**：
- 显示确认框：**"确定要取消上传吗？"**
- 取消后清空已上传的分片记录，释放存储空间
- 用户可重新选择文件开始新的上传

**浏览器刷新/关闭**：
- 上传中断，但服务端保留分片 24 小时
- 用户刷新后，系统自动检测并提示续传

## 三、格式校验与文档类型识别

### 3.1 校验时机与责任分工

| 校验阶段 | 执行位置 | 校验内容 | 触发条件 |
|---------|--------|--------|--------|
| **前置校验** | 前端 JS | 文件扩展名、大小 | 用户选择文件后立即执行 |
| **后端校验** | 后端 API | 文件头（Magic Number）、格式完整性 | 文件上传完成后，进入解析前 |

#### 前端校验逻辑

```
用户选择文件
  ↓
检查扩展名是否在支持列表中？
  ├─ 否 → 提示"不支持的格式"，停止上传
  └─ 是 → 检查文件大小
    ├─ > 50MB → 提示"文件过大"，停止上传
    └─ ≤ 50MB → 发送到后端
      ↓
  后端接收并保存到临时存储
```

**前端不做的事情**：
- ❌ 不尝试打开 PDF/DOCX 文件验证内容（这是后端的职责）
- ❌ 不做任何的格式深度检测（Magic Number 检测）

#### 后端校验逻辑

```
接收文件上传完成信号
  ↓
读取文件头（前 4-16 字节）
  ├─ PDF 文件头：%PDF-1.x → 格式有效
  ├─ DOCX 文件头：PK（ZIP） → 格式有效
  ├─ DOC 文件头：D0CF → 格式有效
  └─ 其他 → 格式错误，返回 400，提示用户
    ↓
如果格式有效，继续文档类型识别流程
```

---

### 3.2 文档类型自动识别

#### 识别机制

**识别时机**：文件上传并通过格式校验后，立即在后端进行

**识别策略**：
1. **规则匹配**（第一层，<100ms）：根据文本内容的关键词进行快速分类
   - 采购合同：包含"商品"/"采购"/"供应商"/"交货日期"
   - 劳动合同：包含"劳动关系"/"员工"/"试用期"
   - NDA：包含"保密"/"机密"/"不得泄露"
   - SaaS 协议：包含"软件服务"/"API"/"SLA"/"数据安全"
   - 其他：无法匹配或匹配到多个类别

2. **LLM 深度识别**（第二层，可选，仅在规则无法确定时）：
   - 输入：文档前 5000 tokens 的纯文本
   - 输出：`{type: "...", confidence: 0.XX}`
   - 仅当 confidence ≥ 0.7 时才接受 LLM 结果

#### 支持的文档类型清单

| 文档类型代码 | 显示名称 | MVP 支持 | 审核规则集 |
|-----------|---------|--------|----------|
| `purchase_contract` | 采购合同 | ✅ | 条款完整性检查、数字字段提取、交付期限验证 |
| `nda` | NDA / 保密协议 | ✅ | 保密范围检查、违约责任上限、执行期限 |
| `labor_contract` | 劳动合同 | ✅ | 试用期规范性、竞业禁止合法性检查、保险覆盖 |
| `saas_agreement` | SaaS 服务协议 | ✅ | SLA 条款完整性、数据安全条款、免责范围 |
| `unknown` | 未知类型 | ✅ | 强制触发人工审核 |

#### 类型识别失败的处理路径

**情景**：系统无法识别文档类型（规则匹配失败，LLM confidence < 0.7）

**处理流程**：
```
类型识别失败
  ↓
状态转换：uploaded → parsing（带 type=unknown 标记）
  ↓
跳过规则引擎审核层（Layer 2）
  ↓
直接进入 LLM 深度分析（Layer 3）
  ↓
同时触发 HITL：状态转换为 human_reviewing
  ↓
律师/法务人工确认文档类型 + 补充审核指示
```

**用户侧提示**：
- 状态进度显示："文档类型未自动识别，已转人工审核"
- 提示用户可在批注中说明文档类型或补充审核需求

### 3.3 不支持文档类型的拦截提示

#### 融资、股权、上下游关联合同拦截

| 文档特征 | 关键词示例 | 拦截提示 |
|---------|----------|--------|
| **融资/股权文件** | "融资"、"股权"、"股份"、"稀释"、"清算优先权"、"融资轮次" | ❌ 该文档类型（融资/股权文件）超出当前系统审核范围，需要专业律师人工审查。建议联系法律顾问。 |
| **合作框架与下游关联** | "框架协议"+"后续"、"本协议项下的具体合同"、"下游供应商" | ⚠️ 系统检测到该协议涉及下游合同联动审查，当前版本不支持跨合同冲突分析。建议人工审查。 |
| **跨司法管辖区竞业禁止** | "竞业禁止"+"美国"/"英国"等地名 | ⚠️ 系统检测到该条款涉及多地司法管辖，AI 无法可靠判断有效性，建议在本地法律顾问确认。 |

**拦截交互**：
- 显示警告框（黄色背景）：不是错误，而是功能边界提示
- 用户可选择：**"忽视此警告，继续上传"** 或 **"返回修改文件"**
- 选择"继续"后，该文档进入 HITL 节点（不经过自动审核）

## 四、文档解析流程

### 4.1 多引擎分层策略

#### 三层解析策略

```
Layer 1: 直接文本提取（< 0.5 秒）
  ↓
  成功？ → 输出纯文本 → 进入审核流程
  失败？ ↓
  
Layer 2: OCR 解析（3-10 秒）
  ├─ 使用 PaddleOCR (中文) 或 Tesseract (英文)
  ├─ 成功？ → 输出 OCR 文本 → 进入质量门控检测
  └─ 失败？ ↓
  
Layer 3: 降级人工处理（SLA: 24 小时）
  ├─ 转移到人工审核队列
  ├─ 人工输入核心信息（或重新上传清晰扫描件）
  └─ 标记为 "人工录入" 的条款，置信度降级
```

#### Layer 1 - 直接文本提取

**适用场景**：原生 PDF（含可搜索文本）、DOCX 文件

**实现方案**：
- PDF：使用 `PyMuPDF` 或 `pdfplumber` 的 `.get_text()` 方法
- DOCX：使用 `python-docx` 的 `document.paragraphs` 迭代提取

**成功标准**：
- 提取的文本字符数 > 原文件预期字符数的 80%（用文件大小推算）
- 提取的文本包含至少 5 个中文/英文句子，证明是真实合同文本（非乱码）

**失败触发**：
- 提取文本为空或字符数 < 20% 阈值 → 升级 Layer 2
- 提取的全是乱码（Unicode 控制字符占比 > 30%） → 升级 Layer 2

#### Layer 2 - OCR 解析

**适用场景**：扫描版 PDF、图片上传

**实现方案**：
- **中文扫描件**：PaddleOCR（精度 95%+，速度快）
- **英文扫描件**：Google Cloud Vision API 或本地 Tesseract（备选）
- **混合文档**：自动检测语言，选择对应引擎

**处理流程**：
```
输入：扫描版 PDF（每页转成图像）
  ↓
逐页 OCR 识别
  ├─ 保存页码信息（用于后续定位）
  ├─ 并行处理最多 3 页（提升速度）
  └─ 页级错误隔离（单页失败不影响其他页）
  ↓
输出：`{page: 1, text: "...", conf_score: 0.92}`
```

**成功标准**：
- 至少 80% 的页面 OCR 识别率 ≥ 80%
- （详见 4.2 质量门控标准）

**失败触发**：
- 超过 20% 的页面无法识别（图片质量太差、非文档图片等）
- 整体识别结果置信度 < 60% → 升级 Layer 3

#### Layer 3 - 降级人工处理

**触发条件**：
- Layer 1、2 均失败，或 Layer 2 输出质量过低
- 系统自动转移到人工队列

**处理时间**：SLA 24 小时（工作日）

**人工处理方式**：
1. 人工阅读文档，手工输入核心字段（合同主体、金额、期限）
2. 或者，请用户上传更清晰的扫描版本
3. 或者，请用户直接上传 Word 版本（如有）

**标记机制**：
- 人工输入的信息标记为 `source: human_input`
- 对应的置信度字段强制设置为 `0.5`（低置信度）
- 审核报告中显示该条款为"人工录入"，需律师复核

---

### 4.2 OCR 质量门控

#### 质量评分标准

当文档进入 OCR 解析后，系统需对 OCR 输出进行质量评估。

**质量评分维度**：

| 维度 | 权重 | 评分规则 | 示例 |
|------|------|--------|------|
| **识别完整性** | 40% | OCR 识别的文本字符数 / 预期字符数（估算） | 预期 5000 字，识别 4500 字 → 90% |
| **单页平均置信度** | 40% | OCR 引擎输出的字符级置信度平均值 | Paddle 平均 conf=0.88 → 88% |
| **乱码率** | 20% | （Unicode 控制字符 + 不可读序列字符数）/ 总字符数 | 乱码 200 字 / 5000 字 → 4% 乱码，质分 96% |

**综合质量得分公式**：
```
quality_score = 0.4 × completion_ratio + 0.4 × avg_confidence + 0.2 × (100 - garble_rate)
```

#### 质量门控阈值

| 质量评分 | 判定 | 处理方式 |
|---------|------|--------|
| ≥ 85% | 高质量 | ✅ 直接通过，进入审核流程 |
| 70-84% | 中等质量 | ⚠️ 通过，但在审核报告中标记"OCR 质量中等，请人工复核数字字段" |
| 60-69% | 低质量 | ❌ 不通过，触发 Layer 3 降级人工；提示用户重新上传 |
| < 60% | 无法识别 | ❌ 文件无法处理，返回错误，提示用户上传清晰版本 |

#### 质量不通过的处理路径

```
OCR 完成，质分 < 70%
  ↓
状态：parsing → parse_failed
  ↓
用户通知："文档质量不清晰，OCR 识别失败。请上传更清晰的版本或 Word 格式。"
  ↓
用户选择：
  ├─ A) 重新上传清晰版本 → 重新进入解析流程
  ├─ B) 直接进入人工审核 → 状态转为 human_reviewing，人工录入关键字段
  └─ C) 放弃这份文档 → 删除记录
```

#### 质量评分的可视化展示

在解析完成后，用户在审核结果页面可看到：

```
OCR 质量评分：88/100 ✅

├─ 文本识别完整性：92%
├─ 字符识别置信度：85%
└─ 乱码率：3%

此文档识别质量良好，可直接审核。
```

### 4.3 WebSocket 实时进度推送设计

#### 推送阶段划分

整个上传→解析→审核过程分为 **5 个阶段**，每个阶段有对应的 WebSocket 消息。

| 阶段编号 | 阶段名称 | 进度占比 | 事件消息 | 持续时间 |
|---------|--------|--------|--------|--------|
| 1 | 上传中 | 0-30% | `upload_progress` | 取决于文件大小和网速 |
| 2 | 解析中 | 30-60% | `parsing_progress` | Layer 1 (<0.5s) + Layer 2 (3-10s) |
| 3 | 质量检测 | 60-70% | `quality_check_progress` | <1s |
| 4 | 审核中 | 70-95% | `reviewing_progress` | 3-30s（取决于审核深度） |
| 5 | 完成 | 95-100% | `completed` / `failed` | - |

#### WebSocket 消息格式

**消息类型：Progress Update**

```json
{
  "event_type": "upload_progress",
  "task_id": "task_abc123",
  "stage": "parsing",
  "stage_number": 2,
  "overall_progress": 45,
  "stage_progress": 60,
  "message": "正在解析文档（Layer 1 文本提取成功，进入 Layer 2）",
  "timestamp": "2026-04-14T10:30:45Z"
}
```

**字段说明**：
- `event_type`：事件类型（用于前端路由）
- `task_id`：该上传任务的唯一 ID（用于绑定前端 UI）
- `stage`：当前阶段名称（upload / parsing / quality_check / reviewing）
- `stage_number`：阶段序号（1-5）
- `overall_progress`：整体进度百分比（0-100）
- `stage_progress`：当前阶段内的进度百分比（0-100）
- `message`：用户友好的状态文案

#### 推送频率规范

| 阶段 | 推送频率 | 说明 |
|------|--------|------|
| 上传中 | 每上传 512KB 一次 | 约 0.5-2 秒一次（取决于网速） |
| 解析中 | 每识别完 1 页一次（PDF） | 约 1-3 秒一次 |
| 质量检测 | 完成时一次 | <1 秒，无需频繁推送 |
| 审核中 | 每处理完一个审核子流程一次 | 规则检查、LLM 分析各一次；约 10-20 秒 |

#### 前端接收与展示逻辑

```javascript
// 前端 WebSocket 连接
const ws = new WebSocket(`wss://api.example.com/ws/task/${taskId}`);

ws.onmessage = (event) => {
  const message = JSON.parse(event.data);
  
  if (message.event_type === 'upload_progress') {
    updateUploadBar(message.overall_progress);
    updateStatusText(message.message);
  } else if (message.event_type === 'parsing_progress') {
    updateProgressBar(message.overall_progress);
    updateStatusText(`${message.stage}: ${message.stage_progress}%`);
  }
  // ... 其他消息类型处理
};
```

#### 网络异常处理

**WebSocket 连接断开**：
- 客户端自动重连，重连策略：1s → 2s → 4s → 8s（指数退避，最多 5 次）
- 重连成功后，服务端发送最新的进度状态（避免状态丢失）
- 用户界面提示"连接已恢复"，继续显示进度

**长时间无推送**（>30 秒）：
- 前端检测超时，显示"处理中，请耐心等待..."
- 允许用户点击"刷新状态"手动查询最新状态

### 4.4 单页失败与整批处理的关系

#### 单页级别的错误隔离

**场景**：用户批量上传 10 个 PDF 文件，其中某个 PDF 的第 5 页无法识别

**处理原则**：
- **页级隔离**：单页失败不影响该 PDF 的其他页面继续处理
- **文件隔离**：该 PDF 的失败不影响其他 9 个 PDF 的处理

#### 处理流程

```
批量上传：file1.pdf, file2.pdf, ... file10.pdf
  ↓
并行处理（最多 3 个并发）
  ├─ file1.pdf
  │   ├─ page 1 ✅ 成功
  │   ├─ page 2 ✅ 成功
  │   └─ page 3 ❌ 失败（质分 < 60%）
  │       └─ 标记 page 3 为 "manual_input_required"
  │       └─ 继续处理其他页面
  │   结果：file1 整体状态 = "partial_fail"
  │
  ├─ file2.pdf
  │   ├─ page 1 ✅ 成功
  │   └─ page 2 ✅ 成功
  │   结果：file2 整体状态 = "success"
  │
  └─ file3.pdf
      ├─ all pages ❌ 全部失败
      结果：file3 整体状态 = "failed"
        → 转人工队列
```

#### 单文件状态定义

| 状态 | 含义 | 后续处理 |
|------|------|--------|
| `success` | 所有页面解析成功 | 进入自动审核（Layer 1-3） |
| `partial_fail` | 部分页面失败，但核心内容识别成功（>80%） | 标注失败页面，进入审核；审核时对失败页面降低置信度 |
| `failed` | 无法解析（质分 < 60% 或无法识别的格式） | 转移到人工审核队列 |

#### 用户侧批量处理视图

```
批量上传结果：

✅ 7/10 个文件已解析
├─ file1.pdf [success] ✅
├─ file2.pdf [partial_fail] ⚠️ （第 3 页需人工复核）
├─ file3.pdf [success] ✅
├─ file4.pdf [failed] ❌ （需要重新上传）
├─ ... 其他文件

用户可以：
- 点击 [failed] 的文件 → 选择重新上传或转人工
- 点击 [partial_fail] 的文件 → 查看详情
- "一键提交审核" → 所有 success/partial_fail 的文件进入审核队列
```

## 五、解析结果保存

### 5.1 元数据保存规范

#### 保存的元数据清单

在文档解析完成后，以下元数据**必须**写入数据库（Document 表和 ReviewTask 表）：

| 元数据字段 | 数据类型 | 来源 | 保存时机 |
|-----------|--------|------|--------|
| `document_id` | UUID | 系统生成 | 文件接收时 |
| `file_name` | String | 用户上传时的原始文件名 | 文件接收时 |
| `file_size` | Integer (bytes) | 操作系统 | 文件接收时 |
| `file_format` | Enum(pdf, docx, doc) | 后端格式校验 | 格式验证通过时 |
| `upload_time` | DateTime | 系统时间 | 文件接收时 |
| `extraction_method` | Enum(direct_text, ocr, manual_input) | 解析引擎 | 解析完成时 |
| `text_content` | LongText | Layer 1/2 输出 | 解析成功时 |
| `ocr_quality_score` | Float(0-100) | OCR 质量评分算法 | OCR 完成时 |
| `document_type` | Enum(purchase, nda, labor, saas, unknown) | 规则匹配或 LLM 识别 | 类型识别完成时 |
| `document_type_confidence` | Float(0-1) | LLM 或规则匹配置信度 | 类型识别完成时 |
| `language` | Enum(zh, en, mixed) | 自动检测 | 解析完成时 |
| `page_count` | Integer | PDF 元数据或 DOCX 段落数 | 解析完成时 |
| `vector_db_version` | String | 系统配置 | 审核开始时 |
| `parsing_status` | Enum(success, partial_fail, failed) | 解析引擎输出 | 解析完成时 |
| `parsing_error_log` | JSON | 解析过程中的错误信息 | 解析失败时 |

#### 保存时机

**Time Point 1: 文件上传完成**
```
document_id, file_name, file_size, upload_time → 写入 Document 表
status = "uploaded"
```

**Time Point 2: 格式校验通过**
```
file_format → 更新 Document 表
status = "uploaded" （不改变）
```

**Time Point 3: 解析完成**
```
text_content, extraction_method, ocr_quality_score, 
document_type, document_type_confidence, language, page_count
→ 更新 Document 表

同时创建或更新 ReviewTask 表：
status = "parsed"
parsing_status = "success" / "partial_fail" / "failed"
```

**Time Point 4: 审核开始**
```
vector_db_version → 写入 ReviewTask 表
status = "auto_reviewing"
```

### 5.2 状态流转逻辑

#### 完整状态机定义

```
[uploaded] 
  ↓ 用户文件接收
[parsing]
  ├─ Layer 1/2/3 解析中
  ↓
[parsed] 或 [parse_failed]
  ├─ 若 parsed：自动进入 auto_reviewing
  └─ 若 parse_failed：等待用户重新上传或人工处理
  
[auto_reviewing]
  ├─ 规则检查（Layer 2）
  ├─ LLM 分析（Layer 3）
  ├─ 触发 HITL 判断
  ↓
[auto_reviewed]
  ├─ 若不触发 HITL → [completed]
  └─ 若触发 HITL → [human_reviewing]
  
[human_reviewing]
  ├─ 律师/法务人工审核
  ├─ 可确认/驳回 AI 结论
  ↓
[completed]

任意状态 → [rejected]（用户手动取消或系统强制取消）
```

#### 状态流转的触发条件

| 源状态 | 目标状态 | 触发条件 | 系统行为 |
|--------|--------|--------|--------|
| `uploaded` | `parsing` | 文件上传完成且格式校验通过 | 后端自动启动解析队列任务 |
| `parsing` | `parsed` | Layer 1/2/3 任意一层成功 | 标记 `extraction_method` 和质量分 |
| `parsing` | `parse_failed` | 所有 Layer 均失败或质分 < 60% | 记录错误日志，通知用户 |
| `parsed` | `auto_reviewing` | 自动转换（无需等待） | 立即进入规则检查队列 |
| `auto_reviewing` | `auto_reviewed` | 规则检查和 LLM 分析完成 | 记录风险列表和置信度 |
| `auto_reviewed` | `completed` | 未触发 HITL 条件 | 生成审核报告，通知用户 |
| `auto_reviewed` | `human_reviewing` | 触发 HITL（见 4.1 场景 C） | 分配给审核人，状态推送通知 |
| `human_reviewing` | `completed` | 审核人完成确认/驳回 | 记录人工审核结果，生成最终报告 |
| 任意 | `rejected` | 用户手动取消 / 系统管理员强制取消 | 清空临时数据，记录取消原因 |

#### 状态变更的日志记录

**每次状态转换都必须记录**：

```json
{
  "task_id": "task_abc123",
  "from_status": "parsing",
  "to_status": "parsed",
  "timestamp": "2026-04-14T10:35:22Z",
  "triggered_by": "system:auto",
  "details": {
    "extraction_method": "ocr",
    "ocr_quality_score": 88.5,
    "document_type": "purchase_contract",
    "page_count": 12
  },
  "remarks": "Layer 2 OCR 解析成功，质分 88.5% 通过"
}
```

此日志保存在 `ReviewTask.status_change_log` 字段（JSON Array），支持完整的审计追溯。

## 七、交互设计关键决策理由

### 7.1 为什么采用多引擎降级策略？

**业界现状**：ABBYY、AWS Textract、合合信息等行业头部产品均采用此策略

**核心理由**：
1. **覆盖率最大化**：直取 + OCR + 人工，可处理 99% 的合同文档
2. **成本控制**：Layer 1 零成本；Layer 2 按页付费；Layer 3 按人时计费，仅用于真正困难的文件
3. **速度与质量平衡**：快速路径（直取）处理 80% 的文件，关键路径（OCR）处理边缘情况

### 7.2 为什么 OCR 必须有质量门控？

**痛点**：扫描件的 OCR 结果质量不稳定，直接输入 LLM 会导致审核结论错误，甚至法律责任

**门控标准设置**（85% 通过阈值）：
- 低于 85% 的 OCR 输出进入人工审核，而非直接参与审核
- 70-84% 的文件仍可处理，但在结果中标记"需人工复核"
- < 70% 的文件必须重新上传或转人工

**业界对标**：Google Document AI、AWS Textract 均提供 confidence score，用户可自定义阈值

### 7.3 为什么用 WebSocket 而不是长轮询？

| 方案 | 延迟 | 服务器负载 | 实施复杂度 |
|------|------|---------|---------|
| WebSocket | 实时 (< 100ms) | 低（持久连接） | 中等（需要连接管理） |
| 长轮询 | 2-5 秒 | 高（频繁请求） | 低（无状态） |
| Server-Sent Events (SSE) | 实时 | 中等 | 中等 |

**为什么选 WebSocket**：
- 法务场景用户等待时间短（解析 10-30 秒）
- 实时反馈能显著提升用户体验
- 支持双向通信（后续可用于用户催办、人工审核人的任务分配）

### 7.4 为什么单页失败不影响整批处理？

**用户价值**：
- 一个 PDF 的一页质量差，不应该让整个文件无法审核
- 允许用户"先用，后补"（先审核可用页面，失败页面人工补录）

**技术可行性**：
- PDF 是页级格式，页间独立
- OCR 本质上也是页级处理
- 后续审核时可标记"第 3 页为人工输入"，降低置信度
